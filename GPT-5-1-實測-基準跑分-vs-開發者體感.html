GPT 5.1 實測：基準跑分 vs. 開發者體感

📝 核心摘要 (已根據公開資料修正)

  • GPT 5.1 在多項基準測試（如 Software Engineering Benchmark (SWEBench)）上超越對手，展現更高精度與驚人的成本效益。
  • 然而，大量開發者回報其效能極不穩定（Non-deterministic），尤其在處理程式碼時。
  • 在特定任務（如 User Interface (UI) 生成、複雜任務規劃）上表現出色，但在程式碼編輯與工具呼叫上卻出現個案回報的難以理解行為。
  • 模型家族（Codex, Mini, 高推理模式）的表現各異，突顯了非預期的效能差異，且效能與模型大小並非總是正相關。

📊 亮眼的基準測試 (含資料修正)

GPT 5.1 在發布時，多項基準測試資料十分亮眼。例如，在 SWEBench 上，GPT 5.1 Codex 版本的得分（70.4%）略高於 Sonnet 4.5（69.8%）（此資料來自第三方 Vals.ai 測試，非官方唯一標準）。

真正的差異在於成本和速度：根據 Vals.ai 的 SWE-bench 測試資料，GPT 5.1 Codex 的執行成本（約 $0.31/test）遠低於 Sonnet 4.5（約 $8.26/test）。

相較之下，它比 GPT-5 Codex（約 $2.21/test）便宜約 7 倍（非 10 倍），並且比 Sonnet 4.5 便宜了驚人的 26 倍之多。（原影片中 $8.26 vs $31 的資料為明顯錯誤）

此外，GPT 5.1 在「人工分析智慧指數」上正確提升了 2 分（68 → 70）。模型展現了更強的可變推理 (Variable Reasoning) 或官方所稱的 Adaptive Reasoning 能力，能根據任務的簡易程度自動調整 Token 的使用量。

🔧 API 新功能與改進

除了原始效能，GPT 5.1 帶來了幾項對開發者極為友善的 應用程式介面 (API) 更新，這些資訊大致正確：

  • 無推理模式 (`reasoning_effort = 'none'`) : OpenAI 官方確認新增了此參數，取代了原先的「Non-reasoning effort option」說法，用於低延遲場景。
  • 擴展提示快取 (Extended Prompt Caching): 快取時間從幾分鐘大幅延長至 24 小時。這對 API 使用者來說是巨大的改進，能顯著降低成本並提升重復查詢的速度。
  • 寫作風格優化 (主觀): 根據主觀使用體驗，透過 API 呼叫時，模型的寫作風格明顯優於 ChatGPT 網站版本。回應更為簡潔、直接（此點無法客觀驗證）。

🤯 編碼實戰：不穩定的工具使用 (個案回報)

儘管基準測試亮眼，但在實際的編碼情境中，GPT 5.1（特別是 Codex 版本）的表現卻充滿了開發者的主觀回報與個案問題。模型在理解與執行工具呼叫時，出現了多種奇怪且低效的行為：

  • 錯誤的工具選擇: 儘管專案中存在 bun.lockb 鎖定文件，模型仍堅持使用 npm 來安裝依賴。
  • 詭異的編輯方式 (個案回報): 最令人費解的是，據稱模型不使用標準的檔案系統操作來修改程式碼，而是轉而使用 Perl 和 regex 執行程式碼變更。這種方法極其荒謬且效率低下。
  • 無意義的指令: 在 IDE (Integrated Development Environment) 環境中，模型會反覆執行 echo 'noop' > /dev/null 這樣毫無意義的空操作命令（此行為有其他開發者獨立回報）。
  • 行程處理失敗: 模型無法正確處理需要長時間運行的背景指令。例如，它會啟動 dev server，然後因為指令未「終止」而無限期掛起。

📈 複雜任務：規劃與執行的落差

GPT 5.1 的表現在不同模型和任務間存在巨大差異。在一次測試中，任務是將一個專案從 Artificial Intelligence Software Development Kit (AI SDK) v4 遷移到 v5（這確實是一次破壞性極大的重構）。

  • Codex 版本的失敗: GPT 5.1 Codex 提出了一個 3 步驟計畫，但在執行時很快就遇到了錯誤，並且無法根據錯誤訊息進行修復。
  • 高推理模式的成功: 相比之下，GPT 5.1 (高推理模式) 版本展現了卓越的規劃能力。它生成了一個詳細且連貫的 7 步驟計畫。隨後，它僅用了兩個主要步驟（規劃與執行）就成功完成了這次極其繁瑣的升級。

這表明，GPT 5.1 的非 Codex 版本（特別是高推理模式）在高層次的「規劃」階段可能遠比其「執行」階段更可靠和智慧。（註：「High」為推理強度參數，非獨立模型）

🎲 極端的非確定性 (個案測試)

GPT 5.1 最大的問題是其極端的非確定性。以下為影片作者的個人測試案例，雖無法視為普遍客觀事實，但突顯了模型行為的不可預測性：

  1. Tokenizer UI 任務 (個案測試):

    任務是建立一個能突顯 Token 邊界的 UI。同時對 7 個不同的模型發出相同指令。結果：

    幾乎所有頂級模型（包括 5.1 Codex, 5.1 (高推理模式)）都失敗了——它們要麼導致頁面崩潰、要麼導入了不存在的模組，要麼完全誤解了需求。
    令人震驚的是，在此特定測試中，唯一完美完成任務的，是 Anthropic 最便宜的 Haiku 4.5 模型。

  2. Tailwind V4 修復任務 (個案測試):

    任務是修復一個損壞的 Tailwind V4 設定。在第一輪測試中，所有模型都失敗了。

    在第二輪測試中，加入了額外提示「請確保使用 Tailwind V4」。結果：
    GPT 5.1、5.1 (高推理模式) 和其他模型仍然失敗。只有 GPT 5.1 Codex 這次正確理解了提示，並成功修復了專案。

這兩項個案測試表明，模型的效能極不穩定。有時，最昂貴的模型會失敗，而最便宜的模型會成功。這使得基準測試的參考價值大打折扣。

🔚 結論：充滿矛盾的強大模型

GPT 5.1 是一個充滿矛盾的集合體。它在特定基準測試（如 SWE-bench）和成本效益上樹立了新標竿，但在實際開發中，卻暴露出大量開發者回報的非確定性和令人費解的工具使用缺陷。

對於開發者而言，它在「規劃」方面的潛力（特別是 5.1 (高推理模式)）可能大於「執行」。而一套新的工作流程——例如使用 5.1 (高推理模式) 進行規劃，再交由 Codex Mini 或 Composer 等高速模型執行——可能是未來更務實的選擇。
