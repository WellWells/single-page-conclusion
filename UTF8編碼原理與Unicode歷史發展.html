🔢 數位計算的起源：從純數字到文字顯示

在 1940 年代，第一代數位計算機的研發主要是由軍事需求驅動，其核心目的在於加速大規模的數值運算，例如計算炮彈軌跡表。因此，當時的硬體結構僅能儲存與處理純數字，運算邏輯也僅限於基礎算術。

隨著二戰結束，程式設計師開始探索數值計算之外的應用，文字的儲存與處理逐漸成為文字編輯、政府紀錄與通訊的核心。然而，即便硬體技術不斷演進，現代電腦在底層依然僅能處理數字。

解決方案： 透過建立對應表（Code Book），為每個英文字母指定一個唯一的數字編號。例如，將 "HELLO" 轉換為「8 5 12 12 15」，這便是文字編碼的初步構想。

📜 ASCII 的誕生與全球化挑戰

American Standard Code for Information Interchange （ASCII）

  • 於 1963 年發布，採用固定寬度（Fixed-width）編碼。
  • 使用 7 個位元（bits），共提供 128 個編碼組合。
  • 涵蓋英文大小寫、數字 0-9、標點符號及 32 個控制碼（如：換行符、水平定位符）。
  • 優點： 在網路極慢的年代，7 位元傳輸速度比 8 位元快 14%，且第 8 位元可用於同位元（Parity bit）偵錯。

進入 1980 年代，電腦普及至非英語系國家。亞洲語言如中文或日文擁有數萬個獨特符號，遠遠超過 ASCII 7 位元所能承載的範圍。

為了解決編碼混亂，各大科技公司聯合提出 Universal Character Encoding Standard （Unicode）。其目標是為人類語言中的每個視覺字元指派唯一的碼點（Code Point）。

核心難題： 既然碼點高達數萬個，勢必需要超過 1 個位元組來儲存，這將導致舊有的 ASCII 軟體完全失效，並使原本的純英文文件體積翻倍。

🧠 鬼才 Ken Thompson 的設計：自同步變長編碼

1992 年，Ken Thompson 在餐巾紙上設計出了 UTF-8，其中最天才的設計在於對「首位元（First Bit）」的巧妙運用。ASCII 只用了 7 個位元，第 8 個位元通常被閒置。UTF-8 正是利用這第 8 個位元（最高位元）作為「切換開關」：

🔍 0 與 1 的藝術：

  • 首位元為 0： 完美對應 ASCII。
    這保證了所有舊有的系統、檔案與程式碼能無痛過渡，是實現全球標準化的關鍵鑰匙。
  • 首位元為 1： 啟動變長編碼機制。
    電腦只要看到 1，就數數看有幾個連在一起的 1，便能立刻知道字元長度（例如 1110... = 3 bytes）。
  • 這種利用首位元區分純 ASCII 與多位元組的設計，同時解決了向後相容與自同步問題，是整個 UTF-8 協議中最聰明、最優雅的細節。

🇰🇷 韓文 Hangul：Unicode 系統中的邏輯實務

結合「字母邏輯」與「方塊結構」的智慧

韓文（Hangul）在 Unicode 編碼系統中展現了極其獨特且聰明的設計。雖然外觀上呈現方塊狀，但其底層邏輯卻是首創的特徵音位文字。這種「形似漢字、神似拼音」的特性，讓電腦科學家為其設計了一套絕妙的編碼方案。

Unique Structure

字母拼裝邏輯

韓文僅由 24 個核心字母（14 輔音、10 元音）組成。與英文不同的是，這些字母在書寫時會依規則組裝成「音節方塊」，在 Unicode 中同時保留了字母編碼與預組音節編碼的雙重彈性。

Space Efficiency

三位元組極限壓縮

若以 ZWJ 方式拼合，一個韓文字可能佔用 20+ Bytes。Unicode 為此預先分配了 11,172 個組合音節，確保在 UTF-8 下，任何常用韓文音節都固定只佔 3 Bytes，大幅優化了亞洲文字的儲存效率。

Mathematical Beauty

免查表的數學映射

韓文編碼並非隨機排列，而是遵循嚴謹的線性代數公式。程式無需依賴龐大的查表（Lookup Table），只要透過簡單的數學運算，就能瞬間完成音節的組合與拆解。

🧩 ZWJ：Unicode 的字元組合機制 U+200D

零寬度連接符（Zero Width Joiner） 是隱藏在現代文字背後的「幽靈字元」。它本身不顯示任何形狀，但卻能指揮渲染引擎將前後的獨立字元「融合」成一個全新的視覺符號。

⚙️ 運作原理：家庭 Emoji 拆解

👨 + ZWJ + 👩 + ZWJ + 👧 = 👨‍👩‍👧

透過 ZWJ 連接，三個獨立的「人」被組合成一個「家庭」。若系統不支援 ZWJ，會自動以相容模式降級顯示為三個分開顯示的圖示。

🤔 技術回顧：UTF-16 與 UTF-32 的現狀

這是一個非常銳利的技術觀察。雖然在網際網路傳輸與儲存領域，UTF-8 已經幾乎成為事實上的主流標準（>99% 占有率），但在系統內部運作與記憶體處理層面，UTF-16 與 UTF-32 依然扮演著不可或缺的角色。簡單來說： 「UTF-8 贏了傳輸，但 UTF-16 依然佔據著許多作業系統的核心。」

16

UTF-16：系統記憶體的關鍵架構

  • 底層基石 Windows API（NT 核心）、Java JVM、JavaScript（V8 引擎）、C# .NET 在記憶體中處理字串時，預設核心依然是 UTF-16。
  • 歷史包袱與優勢 在 90 年代，UCS-2（2 Bytes）被認為足夠涵蓋所有文字，因此被大量系統採納。至今為了相容性（如 Windows 的 `wchar_t`），這個架構被保留下來。
  • 亞洲文字效率 對於純 CJK（中日韓）環境，UTF-16 常用字僅需 2 Bytes，相比 UTF-8 的 3 Bytes，在記憶體資料庫中可能有 33% 的空間優勢。
32

UTF-32：演算法處理的極致利基

  • 真正的 Fixed-width 唯一「碼點 = 記憶體單位」的編碼。可以透過陣列索引 `String[N]` 以 O(1) 時間瞬間存取任意字元。
  • 使用場景 極限於科算、複雜文字處理引擎內部。當需要頻繁隨機跳轉讀取時，會先轉為 UTF-32 處理，完畢後再轉回 UTF-8 儲存。
  • 致命缺點 極度浪費空間（是 ASCII 的 4 倍），且有 Endianness（BOM）問題，完全不適合網路傳輸。
