隨著軟體功能膨脹，我們如何一步步將雜亂無章的程式碼重構成工業級的 RTOS 架構？

階段一 硬體使用率低 / 極易相互阻塞

Delay-based Superloop (阻塞型超迴圈)

這是初學者最常用的方式。所有邏輯循序執行，利用 delay() 或 sleep() 來控制執行頻率。其致命缺點在於：任何一個任務的延遲都會直接阻塞 CPU，導致其他高時效性任務完全失準。

superloop_delay.c C Code
void main(void) {
    while(1) {
        blink_led();   // 希望 500ms 執行一次
        delay_ms(500); // 致命阻塞！

        update_servo(); // 慘遭延誤，無法達到 100Hz (10ms) 更新率
        delay_ms(10);
    }
}
階段二 非阻塞 / 時間分配改善

Timestamp-based Superloop (時間戳記超迴圈)

為了解決延遲造成的全盤卡死，我們使用硬體提供的高精度滴答計數器（如 millis()）。每個任務維護自己的「上一次執行時間戳記」，只有在時間到期時才執行。這能使多個任務在表面上「並行」運行。

superloop_timestamps.c C Code
void main(void) {
    uint32_t last_led = 0, last_servo = 0;
    while(1) {
        uint32_t now = get_timestamp_ms();
        if (now - last_led >= 500) {
            blink_led();
            last_led = now;
        }
        if (now - last_servo >= 10) {
            update_servo(); // 只要其他任務不卡死，此處能維持 10ms
            last_servo = now;
        }
    }
}
階段三 高度結構化 / 合作約束

Cooperative Scheduler (合作式排程器)

當任務達到十幾個，手動寫 if 變得無法維護。我們定義一個 Task 結構陣列，將函式指標與間隔封裝。主迴圈走訪該陣列，統一進行時序排程。缺點： 一旦有任務不自律（例如傳送慢速 UART 阻塞了 300ms），排程系統無能為力。

cooperative_scheduler.c C Code
typedef struct {
    void (*task_func)(void);
    uint32_t period_ms;
    uint32_t last_run;
} Task;

Task tasks[] = { {blink_led, 500, 0}, {update_servo, 10, 0} };

void main(void) {
    while(1) {
        for(int i = 0; i < NUM_TASKS; i++) {
            uint32_t now = get_timestamp_ms();
            if (now - tasks[i].last_run >= tasks[i].period_ms) {
                tasks[i].task_func(); // 執行任務指標
                tasks[i].last_run = now;
            }
        }
    }
}
階段四 硬體中斷搶佔 / 強時效性保證

Preemptive RTOS (搶佔式即時作業系統)

終極解決方案。引進硬體計時器中斷（SysTick，通常 1ms 一次）。在中斷服務程序中，系統強行中斷目前執行的低優先權任務，將暫存器、堆疊指標壓入其專屬堆疊保存（內文切換），改為執行高優先權任務。任務像是在獨立的 CPU 上平行執行。

free_rtos_example.c C Code
// 每個任務都是一個無盡迴圈，自成一格
void vServoTask(void *pvParameters) {
    TickType_t xLastWakeTime = xTaskGetTickCount();
    while(1) {
        update_servo();
        // 延遲 10ms。在等待期間任務進入 Blocked (阻塞) 狀態，自動讓出 CPU 控制權
        vTaskDelayUntil(&xLastWakeTime, pdMS_TO_TICKS(10));
    }
}

void main(void) {
    // 建立任務並賦予優先權
    xTaskCreate(vServoTask, "Servo", 128, NULL, 3, NULL); // 優先權 3 (高)
    xTaskCreate(vLedTask,   "LED",   64,  NULL, 1, NULL); // 優先權 1 (低)
    vTaskStartScheduler(); // 啟動核心排程器
}

核心底層：搶佔式排程與內文切換 (Context Switch) 剖析

1 硬體中斷與中斷響應

當硬體計合器（例如 ARM Cortex-M 的 SysTick）觸發中斷時，CPU 會立即中止當前指令。硬體會自動將基本暫存器（R0-R3, R12, LR, PC, xPSR）壓入當前任務的私有堆疊 (Task Stack)。

2 軟體保存與 TCB 更新

核心的排程中斷服務程序（如 PendSV）會被喚醒，用組合語言將剩下的暫存器（R4-R11）手動壓入堆疊。隨後，將此任務的堆疊指標（SP）儲存至其對應的 任務控制塊 (TCB, Task Control Block) 中。

3 尋找就緒任務與恢復

排程器演算法在就緒列表中尋找優先權最高的任務。將 CPU 的堆疊指標（SP）指向該新任務的堆疊，逆向彈出（Pop）R4-R11。當執行中斷返回指令（BX LR）時，硬體會自動還原其餘暫存器並跳轉至上次中斷時的 PC，無縫恢復執行。

RTOS 同步與通訊實驗室

當多個任務平行運作時，共享硬體資源（如 UART 周邊）會引發衝突，資料互相交錯產生亂碼。體驗互斥鎖與訊息佇列如何保障執行安全。

並行衝突：Preemptive RTOS 下的臨界資源搶佔

在 Preemptive RTOS (搶佔式作業系統) 中，任務隨時可能在任何指令執行到一半時被強制中斷，並切換至另一個任務。 當多個任務同時呼叫底層 UART 傳送日誌時，由於 UART 傳輸字元需要時間，若任務 A 在輸出尚未結束時被高優先權的任務 B 搶佔，任務 B 隨後也向 UART 寫入資料， 這將導致輸出序列在物理上交織在一起，產生人類與解析器皆無法閱讀的亂碼。這種被多任務共同存取的硬體暫存器或全域變數，我們稱為臨界資源 (Critical Section)。

互斥鎖 (Mutex) 周邊保護

Mutex 保護

當開啟 Mutex 保護後，UART 周邊在同一時間只允許單一任務持有。 未搶佔到 Mutex 的任務會進入 Blocked (阻塞) 狀態，暫時讓出 CPU 控制權，直到持有者釋放 Lock，排程器才會喚醒等待佇列中的下一個任務。

mutex_protection.c FreeRTOS API
// 宣告 Mutex 控制代碼 (Handle)
SemaphoreHandle_t xUartMutex;

void vTask(void *pvParameters) {
    while(1) {
        // 取得 Mutex，無限期等待 (portMAX_DELAY)
        if (xSemaphoreTake(xUartMutex, portMAX_DELAY) == pdTRUE) {
            UART_WriteString("Data Log");
            // 釋放 Mutex 保護
            xSemaphoreGive(xUartMutex);
        }
        vTaskDelay(pdMS_TO_TICKS(100));
    }
}

事件驅動：訊息佇列 (Queue)

解耦 / 非阻塞傳送

直接在 10ms 的控制環中呼叫緩慢的傳送函式（即使受 Mutex 保護）也會迫使高優先權任務停滯。 利用訊息佇列 (Queue) 進行非同步解耦：時效性任務僅需將資料打包塞入緩衝 Queue，隨即返回執行，由專職的低優先權日誌任務在背景負責消耗傳輸。

queue_messaging.c FreeRTOS API
// 宣告 Queue 控制代碼 (Handle)
QueueHandle_t xLogQueue;

// 1. 高優先權傳送任務：非阻塞傳入 (Wait=0)
xQueueSend(xLogQueue, &msg, 0); 

// 2. 低優先權日誌任務：平時睡眠，有資料時才喚醒
void vLogTask(void *pvParameters) {
    LogMsg_t rxMsg;
    while(1) {
        // 佇列無資料時，任務自動進入 Blocked (阻塞) 狀態 (不耗 CPU)
        if (xQueueReceive(xLogQueue, &rxMsg, portMAX_DELAY) == pdTRUE) {
            UART_Write(rxMsg.data);
        }
    }
}

RTOS 不是免費的午餐。它提供了豐富的機制，但也會佔用有限的單晶片內部 SRAM 與 Flash。請依您預計規劃的微控制器專案調整滑桿，估算其記憶體開銷。

RTOS 靜態 SRAM 開銷底層計算原理 (32-bit Cortex-M 物理拆解)

1. TCB 結構估算 (~84 Bytes) 精確解析

在 FreeRTOS 中，每個 Task Control Block (TCB) 都是保存該任務所有管理中繼資料的結構：

  • pxTopOfStack (4 Bytes) 指向私有堆疊頂端
  • xStateListItem (20 Bytes) 鏈結就緒/阻塞表
  • uxPriority (4 Bytes) 存放目前優先權
  • pxStack (4 Bytes) 私有堆疊起點指標
  • pcTaskName (12 Bytes) 任務偵錯用字串
  • uxTCBNumber, uxTaskNumber (8 Bytes)

在 32-bit 系統上，不計選配巨集通常固定佔用約 84 ~ 100 位元組 (Bytes) 的 SRAM。

2. 任務堆疊深度與中斷推疊機制 Stack Frame

為什麼任務堆疊不可太小？因為在 ARM Cortex-M 當中，一個中斷發生或內文切換時，以下暫存器會被自動與手動壓入堆疊 (Push)：

  • 硬體自動推疊 (8 Words / 32 Bytes)：
    xPSR, PC, LR, R12, R3, R2, R1, R0
  • 排程器手動推疊 (8 Words / 32 Bytes)：
    R11, R10, R9, R8, R7, R6, R5, R4
  • 區域變數 & 函式呼叫巢狀深度額外開銷

這代表僅僅一次基本的搶佔，堆疊就必須撥出至少 64 位元組 (16 Words) 的硬體底層緩衝。這也是為何 FreeRTOS configMINIMAL_STACK_SIZE 建議不可低於 128 字組 (512 位元組)。

3. IPC 控制塊開銷 (Queue / Mutex) IPC Control

Queue 與 Mutex 在記憶體管理中不單是簡單陣列，而是由複雜控制結構守護：

  • ● 訊息佇列 (Queue) 計算公式：
    $\text{Total} = \text{Queue Header (~80 B)} + (\text{Length} \times \text{Item Size})$ Header 內含雙向任務阻塞等待清單、環形緩衝指標等。
  • ● 互斥鎖 (Mutex) 計算公式：
    $\text{Total} = \text{Queue Header (~80 B)}$ Mutex 在 FreeRTOS 底層本質上是長度為 1、Item 大小為 0 的特殊 Queue，不佔用 Data Buffer，僅需控制塊開銷，但內含額外的優先權繼承 (Priority Inheritance) 邏輯。

何時該考慮導入 RTOS？

在實際專案開發中，不需要對所有需求都盲目套用 RTOS，因其會帶來額外的學習成本、偵錯 (Debug) 複雜度與空間開銷。在作出決策前，請對照以下四大技術特徵。

  • 多速率非同步任務：系統存在需要以不同精準速率運行的工作（如 100Hz 姿態控制環 + 2Hz 狀態閃燈 + 非同步藍牙通訊）。
  • 任務中包含阻塞 API：依賴第三方緩慢協定堆疊、快閃記憶體寫入或高延遲周邊傳輸，且這些傳輸過程無法輕易利用非阻塞 DMA 徹底消除。
  • 高度依賴非同步事件：專案需要事件觸發，而非不間斷輪詢（Poll）。事件驅動（Event-driven）能大幅降低晶片功耗，有助於通過綠色能效標準。
  • 程式碼簿記 (Bookkeeping) 超出應用層邏輯：若您發現自己花了 60% 以上的時間編寫時間戳記比較、中斷標記狀態機、手動儲存狀態暫存器，那說明您的專案正打算自己重寫一個粗糙的 RTOS 機制，此時直接引入 FreeRTOS 或 Zephyr RTOS 是更佳的技術決策。

主流 RTOS 生態系選型指南

業界常規選型因專案規模而定。下面為目前兩大主流系統的橫向比較：

FreeRTOS 極簡 / 輕量 / 市佔率最高

僅提供純粹的核心排程、號誌 (Semaphore)、Queue、Mutex 與記憶體管理 (Memory Management) 模組。適合直接與廠商底層驅動（HAL）整合，程式碼量極其精巧，非常容易移植至各種小規模晶片，亦是學習 RTOS 首選。
