🔓 為什麼要在本地端跑模型

像 Qwen、Kimi 以及 GLM 系列這樣的開源模型，如今已經強大到「不見得需要託管式 API（Application Programming Interface，API）」的程度。把它們跑在自己的筆電上，就沒有人會看到對話內容或資料——這是隱私與資料自主最直接的好處。

要做到這件事，有五款值得認識的工具。它們的抽象層次不同：有的是貼近硬體的推論引擎，有的是開箱即用的開發者工具或桌面程式，有的則是為了承接大量使用者而生的服務引擎。理解它們各自的定位，比死記指令更重要。

01

llama.cpp · 底層推論引擎

用 C++ 寫成，能在 CPU、GPU 與 Apple 晶片上跑，是整個本地工具生態的地基。

llama.cpp 是一套以 C++ 撰寫的推論引擎，能在中央處理器（Central Processing Unit，CPU）、圖形處理器（Graphics Processing Unit，GPU）以及 Apple 晶片上運行。它最初只是一個「想在 MacBook 上跑 Llama」的業餘專案，後來卻成長為其他多數本地工具賴以建構的基礎。

llama.cpp 也帶來了本地模型的標準檔案格式 GGUF。一個 GGUF 檔把權重、斷詞器（tokenizer）與中繼資料全部打包成單一檔案，並支援把精度量化（quantization）壓到 4-bit 甚至更低——正是這種壓縮，才讓龐大的模型塞得進消費級硬體。

使用流程很直接：從 Hugging Face 下載一個 GGUF 檔，執行 llama.cpp 的命令列工具，指定模型與提示詞，就能拿回生成的 token。

# 從 Hugging Face 下載 GGUF 後，直接餵入模型與提示詞
llama-cli -m ./qwen2.5-7b-instruct-q4_k_m.gguf -p "用一句話解釋量化"

何時選它

想要最輕量的執行環境，或要部署到資源受限的硬體——例如邊緣裝置，或一台沒有獨立 GPU 的筆電時。

02

Ollama · 開發者捷徑

在 llama.cpp 外面包了一層，把它變成開箱即用的開發者工具。

Ollama 是包在 llama.cpp 之外的一層封裝，把它變成一套順手的開發者工具：模型下載、量化選項、啟動本地伺服器，全都幫忙處理好，讓人能直接和任何模型對話。

例如執行 ollama run gemma3，它就會自動拉取權重、啟動本地伺服器，並給出一個對話提示符，這些步驟都不必手動處理。更重要的是，這個伺服器提供與 OpenAI 相容的 API——任何 OpenAI 用戶端函式庫，只要把 base URL 換成本機位址這一行，就能直接沿用。

# 啟動模型（自動下載權重並開伺服器）
ollama run gemma3

# 沿用任何 OpenAI 用戶端，只改一行 base URL
base_url = "http://localhost:11434/v1"

何時選它

想要從挑模型到在程式裡呼叫它的最短路徑。這也是工程師為 AI 系統做原型時，最常見的起點。

03

LM Studio · 圖形介面

不必碰終端機、不必改設定檔，用點選的就能跑模型。

LM Studio 是一款用來運行模型的桌面應用程式，提供圖形使用者介面（Graphical User Interface，GUI）。沒有終端機、沒有設定檔；在 Linux、macOS 或 Windows 上安裝後，在程式內搜尋模型、按下下載、就能開始對話。

它底層同樣是包住 llama.cpp，但外面罩了一層介面：在下載任何東西之前，先讓人看到硬體需求、量化選項與 GPU 卸載（offload）設定。如果某個模型對這台機器來說太大，它會事先提出警告。

LM Studio 是瀏覽與比較模型最輕鬆的方式。可以直接在程式裡逛 Hugging Face、看到每一種可用的量化版本、下載幾個來試，並且不必重啟任何東西就能在它們之間切換——這對於釐清「哪個開源模型最適合自己的硬體與任務」非常有幫助。

何時選它

如果是一般使用者，只想要一個簡單的介面來和模型聊天。

04

vLLM · 正式環境服務引擎

Ollama 為快速原型而生，vLLM 則是為了同時服務大量使用者。

vLLM 是一套為「同時服務眾多使用者」而打造的推論引擎。它面向的是正式環境：在一張或多張 GPU 上跑高吞吐量的推論。它的速度主要來自兩項技術。

PagedAttention

一種記憶體更有效率的注意力（attention）演算法版本。少了它，KV 快取（Key-Value Cache）會被當成一整塊連續記憶體存放，造成浪費。PagedAttention 把 KV 快取切成固定大小的區塊，這些區塊不必在 GPU 記憶體裡連續排列，於是騰出記憶體支撐更大的批次，提高吞吐量與並行度。

連續批次處理（Continuous Batching）

一種為模型服務設計的請求排程技術。沒有它，GPU 必須等同一批次裡每個請求都完成，才能開始下一批。連續批次處理讓新請求一有空位就能加入正在執行的批次。兩項技術合起來，能顯著拉高 GPU 上的吞吐量。

vLLM 正是許多公司在幕後運行的引擎，用於內部聊天機器人、程式設計助理或批次處理流程。

🌿 替代選項：SGLang

vLLM 的一個替代品是 SGLang，出自柏克萊 LMSYS 團隊的高速服務引擎。它採用名為 RadixAttention 的技術，靠一個樹狀結構在多個請求之間快取共用的提示詞前綴。這讓它在前綴常常很長又重複的工作負載上特別快——例如檢索增強生成（Retrieval-Augmented Generation，RAG）與多輪對話。SGLang 是 xAI 與許多 DeepSeek 部署在正式環境採用的引擎。

何時選它

當你已經過了原型階段，需要把本地模型端給真實流量時，用 vLLM 或 SGLang——例如為公司上線聊天機器人、為團隊推出程式助理，或執行大規模的內部批次任務。

05

MLX-LM · 為 Apple Silicon 而生

Apple 自家打造，專門在 M 系列晶片上跑模型。

MLX-LM 是 Apple 製作的工具，用於在搭載 M 系列晶片的裝置上運行模型。它之所以重要，關鍵在於記憶體。

在一般 PC 上，CPU 與 GPU 各有獨立的記憶體，模型必須單獨塞進通常容量不大的 GPU 記憶體裡。但在 M 系列的 Mac 上，CPU 與 GPU 共用同一個大型記憶體池：一台配備 192GB 記憶體的 Mac Studio，能載入在 PC 上得用好幾張昂貴 GPU 才裝得下的模型。

MLX-LM 正是為了把這項優勢發揮到極致而設計。

何時選它

如果你用的是 Apple Silicon 的 Mac，又想要可能的最佳速度，這就是該用的工具。
