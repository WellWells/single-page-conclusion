🚀 效能的巨大飛躍

Tailscale 超過 90% 的連線是點對點（P2P）直接連線，但總有些棘手的網路環境（如雲端供應商的託管 NAT 閘道或嚴格防火牆）讓直接連線變得困難。

以往，這類連線會 fallback 到 Tailscale 的全球 DERP 中繼伺服器。DERP 非常可靠，即使在雙方都處於嚴格防火牆後，它仍能建立中繼連線，但作為一項共享服務，它有 QoS 限制，效能並非其首要目標。

現在，Peer Relays 讓您能在自己的 tailnet 中指定一個節點作為中繼。這能帶來天文數字般的效能提升——10 倍增速輕而易舉，實驗室條件下甚至觀測到數個量級的差異。

🛡️ 可靠性與控制權

效能並非唯一優勢。Peer Relays 同時顯著提升了可靠性與控制權。您不再依賴 Tailscale 提供的共享資源，而是使用您自己網路中、專屬於您的中繼節點。

這對於有合規性、成本或安全考量的私有網路尤其重要。以往在這些 locked-down 環境中實現全網狀（full-mesh）連線可能需要複雜的 Subnet Router 設定。

有了 Peer Relays，客戶可以透過一兩個熟知的端點進行中繼，大幅簡化網路架構，並在受限網路中輕鬆實現全網狀模式。

此外，對於尋求完全自主控制的用戶，傳統上唯一的選擇是使用如 Headscale 這類第三方開源控制伺服器來自建 DERP 叢集。這不僅過程複雜、維運成本高，且效能調校困難。Peer Relays 則提供了一個官方支援、輕量級且高效能的替代方案。

當一台設備嘗試連線到 tailnet 中的另一台設備時，它會優先檢查是否可能建立直接連線。如果不行，它會尋找 tailnet 中可用的 Peer Relays。如果 Peer Relay 可用且可存取，設備將通過 Peer Relay 建立連線。如果沒有可用的 Peer Relays，設備最終將 fallback 到使用 DERP 伺服器來中繼流量。

🛠️ 實際應用場景

☁️ 穿透雲端 NAT 閘道

應用程式（如 AWS Managed NAT Gateway 後的服務）可利用 Peer Relays 中繼無法建立直接連線的流量，獲取近乎本機的存取速度。

🔥 穿越嚴格防火牆

在嚴格防火牆環境中，僅需為 Peer Relay 暴露單一 UDP 埠口，即可安全存取後端網路，大幅簡化防火牆規則與資安審核流程。

🚚 卸載 DERP 流量

最小化流經 Tailscale 託管 DERP 網路的資料量，並移除維護自訂 DERP 伺服器的需求，節省維運成本。

🤝 存取夥伴/客戶網路

資料平面流量可透過夥伴網路中可預測的 Peer Relay 端點進行中繼，對方僅需開放最少的埠口，即可建立可控、可審計且高效能的跨網連線。

⚙️ 如何設定 Peer Relays

📋 先決條件

帳號需求

  • 擁有 Tailscale 帳號並可存取 tailnet。
  • 至少一台設備符合 Peer Relay 節點需求。
  • 具備 Owner、Admin 或 Network admin 權限的帳號。

Peer Relay 節點需求

  • 運作支援的作業系統 (iOS 或 Apple TV 除外)。
  • Tailscale 版本 1.86 或更新。
  • 足夠的網路頻寬與低延遲。
  • 至少一個可用的 UDP 埠口，且 tailnet 中其他設備可存取。

其他 Tailnet 節點需求

  • Tailscale 版本 1.86 或更新。
  • 可存取 Peer Relay 節點 (受 ACL 策略規範)。
  • 可存取 Peer Relay 節點上設定的 UDP 埠口。
  • (使用端節點可以是任何作業系統，包含 iOS 和 Apple TV)

🚀 開始設定三步驟

步驟一：設定節點為 Peer Relay

在您想作為 Peer Relay 的節點上，開啟終端機並執行以下指令，指定一個 UDP 埠口 (例如 40000)：

tailscale set --relay-server-port=40000

步驟二：建立 Peer Relay 策略 (Grant Policy)

設備需要被授權才能使用 Peer Relay。請至 Admin Console 的 Access controls 頁面，在 "grants" 區塊中加入策略。 此策略使用 tailscale.com/cap/relay 能力，定義 "src" (可被中繼存取的節點) 和 "dst" (擔任中繼的 Peer Relay 節點)。

例如，允許 tag:us-east-relays 的節點為 tag:us-east-vpc 的節點中繼流量：

{
  "grants": [
    {
      "src": ["tag:us-east-vpc"],
      "dst": ["tag:us-east-relays"],
      "app": {
        "tailscale.com/cap/relay": []
      }
    }
  ]
}

⚠️ 注意： 避免在 "src" 中使用過於寬鬆的目標 (如 "*")，這可能導致非預期的流量繞行和高延遲。"src" 應指定那些位於嚴格 NAT 後、位置固定的設備 (如雲端 VPC 或公司網路)，而非經常移動的筆電或手機。

步驟三：觀察 Peer Relay 流量

產生一些流量來測試設定。然後，使用 tailscale status 來驗證連線是否正透過 Peer Relay 運作：

tailscale status | grep peer-relay

當連線成功時，您會看到 peer-relay 取代了 direct 或 relay：

<tailscale-ip> <hostname> <user> <os> active; peer-relay <ip>:<port>...

📚 策略範例

範例 1：使用節點自身作為 Relay

在某些情境下 (例如防火牆阻擋了所有 Tailscale 埠口，但您想開放單一 UDP 埠口)，您可以讓節點自我中繼。

{
  "grants": [
    {
      "src": ["us-west-1"],
      "dst": ["us-west-1"],
      "app": {
        "tailscale.com/cap/relay": []
      }
    }
  ]
}

範例 2：存取受限 VPC (搭配 ACL)

當您設定 Peer Relay ("grants") 後，您仍需要 ACL 存取規則 (在 "acls" 區塊) 來允許流量通過。 例如，允許 tag:us-east-database 透過 tag:us-east-relays 存取 tag:us-east-vpc：

(1) 授權 Relay (Grants):

{
  "grants": [
    { "src": ["tag:us-east-vpc"], "dst": ["tag:us-east-relays"], "app": {"tailscale.com/cap/relay": []} }
  ]
}

(2) 允許存取 (ACLs): (此為示意，非 grants 的一部分)

{
  "acls": [
    {
      "action": "accept",
      "src":    ["tag:us-east-database"],
      "dst":    ["tag:us-east-vpc:*"]
    },
    {
      "action": "accept",
      "src":    ["tag:us-east-database"],
      "dst":    ["tag:us-east-relays:*"]
    }
  ]
}

🔒 安全性與存取控制

Peer Relays 僅能為同一 tailnet 中的設備中繼流量，並且同樣受到您的 ACL 存取策略约束。 您為 Peer Relay 設定的 UDP 埠口必須對 tailnet 中其他設備開放。請確保您的防火牆或網路安全設定允許該埠口的傳入流量。所有流量，無論是直接、Peer Relay 還是 DERP，始終都受到 WireGuard® 的點對點加密保護。
