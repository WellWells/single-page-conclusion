🚀 Jenkins 自動化部署實戰攻略

掌握 DevOps（Development and Operations）核心技能，從零建置 CI/CD（Continuous Integration / Continuous Delivery）持續整合與交付流程，整合 Docker 與 Pipeline（流水線）的全方位技術解析。

💡 核心摘要

本指南深入探討 Jenkins 在現代軟體開發週期中的關鍵角色。從基礎的 CI/CD 概念出發，詳細解說如何利用 Docker 快速建置 Jenkins 環境，並透過 Pipeline 技術將程式碼的建置、測試與部署流程自動化。內容涵蓋 Jenkinsfile 的版本控制策略、Git/GitHub 整合、Maven 專案建置，以及利用 SSH（Secure Shell） 實現遠端伺服器自動部署。此外，亦探討了分散式建置架構（Master/Agent），協助工程師打造高效率、可擴展的 DevOps 工作流。

💡 給新手的避開陷阱指南

  • 🛡️
    小心 Docker 權限深水區

    遇到 permission denied 時不要慌張。雖然 chmod 666 docker.sock 可以快速解決問題，但請記住這是不安全的權宜之計，生產環境請務必使用標準 Group 設定。

  • 🔑
    SSH 金鑰的隱形門檻

    觀念上是「鑰匙與鎖頭」，但實作上是「檔案權限」。請將此口訣記在心裡：私鑰檔案必須是 600 權限，.ssh 目錄必須是 700 權限，否則連線會直接被拒絕。

  • 🐛
    擁抱試錯 (Trial and Error)

    第一次撰寫 Jenkinsfile 幾乎 100% 會遇到語法錯誤或路徑問題。這是學習過程的常態，不是你做錯了什麼，請善用 Log 進行除錯。

⚙️ CI/CD 與 DevOps 概論

理解自動化流程的價值，為何現代開發團隊必須導入持續整合與交付。

CI（Continuous Integration）持續整合

開發者頻繁地將程式碼合併至主分支。系統自動觸發編譯、建置與單元測試，儘早發現整合錯誤，避免「整合地獄」。核心價值在於快速回饋與降低風險。

CD（Continuous Delivery/Deployment）持續交付/部署

在 CI 的基礎上，自動將通過測試的程式碼部署至測試環境或生產環境。持續交付確保軟體隨時處於可發布狀態；持續部署則進一步自動化上線流程，減少人為介入。

🐣 給初學者的白話文：餐廳比喻

想像你在經營一家高品質餐廳：
CI (持續整合) 就像是「試菜員」。每當廚師（開發者）發明新菜色，試菜員會立刻檢查味道是否合格（測試），確保不會讓難吃的菜流出去。
CD (持續交付/部署) 就像是「自動上菜系統」。一旦試菜通過，這套系統會自動把菜送到客人的桌上（生產環境），中間不需要服務生手忙腳亂地傳遞。

📦 環境建置與安裝

推薦使用 Docker 進行輕量化部署，解決繁瑣的 Java 環境相依性問題。

系統需求與準備

  • Java 環境：
    • Jenkins 2.357 - 2.462：支援 Java 11, 17, 21
    • Jenkins 2.463+ (2024/06後)：需要 Java 17 或 21 (不支援 Java 11)
  • Docker (推薦)： 使用官方映像檔可快速啟動，並易於管理資料持久化。

Docker 啟動指令

docker run -d \
  -u root \
  --name jenkins \
  -p 8080:8080 \
  -p 50000:50000 \
  -v jenkins_home:/var/jenkins_home \
  -v /var/run/docker.sock:/var/run/docker.sock \
  jenkins/jenkins:lts

參數解析：
-p 8080:8080：Web 介面通訊埠。
-p 50000:50000：Agent 節點通訊埠。
-v jenkins_home:...：掛載資料卷，確保重啟容器後設定與資料不遺失。
-v ...docker.sock：（選用）若需在 Jenkins 內部操作實體主機 Docker 時使用。

💡 為什麼要設定 -v jenkins_home？

Docker 容器就像是一個「用完即丟」的盒子。如果你刪除或者是升級這個容器，裡面所有的資料（使用者帳號、建置歷史、Pipeline 設定）都會瞬間消失！
透過 -v 掛載資料卷，就像是把資料存放在盒子外面的保險箱。即使盒子換了，保險箱裡的資料還在，這是資料持久化的關鍵。

🚨 重要安全警告：Docker Socket 權限風險

當掛載 /var/run/docker.sock 時，Jenkins 容器內的使用者可能無權存取實體主機的 Docker 常駐程式，導致建置失敗。

⚡ 快速解決方案（僅限本地測試/學習環境）：

在實體主機執行以下指令修改權限：

sudo chmod 666 /var/run/docker.sock

⛔ 此方法讓任何容器都能完全控制實體主機 Docker，等同給予 root 權限！生產環境絕對禁用！

✅ 生產環境安全做法：

方法一：建立 Docker 群組

sudo groupadd docker
sudo usermod -aG docker jenkins
sudo systemctl restart docker

方法二：使用 Docker-in-Docker (DinD)

docker run -d --privileged \
  --name jenkins-dind \
  docker:dind

詳細說明請參考：Docker-in-Docker 官方文件

初始化設定

首次啟動需輸入解鎖密碼。若使用 Docker，可透過以下指令獲取： 

docker exec -it jenkins cat /var/jenkins_home/secrets/initialAdminPassword

接著安裝「建議的套件 (Suggested Plugins)」並建立管理員帳戶即可完成初始設定。

📝 Pipeline 與 Jenkinsfile

Pipeline as Code 是現代 CI/CD 的標準。使用 Declarative Syntax（宣告式語法）定義清晰的建置流程。

Freestyle vs. Pipeline

特性 Freestyle Project Pipeline
設定方式 GUI（圖形使用者介面）點擊設定 撰寫腳本程式碼 (Groovy)
版本控制 困難 (存於 XML) 容易 (Jenkinsfile 隨專案版控)
流程複雜度 適合簡單單一任務 支援複雜邏輯、平行處理、暫停重啟

Declarative Pipeline 範例

推薦使用聲明式語法 (Declarative)，結構清晰且易於維護。
Tools」中已設定好名為 Maven 3 與 JDK 17 的工具設定。

📜 Pipeline 語法結構拆解
  • pipeline { ... }：整個自動化流程的最外層容器。
  • agent any：誰來做？ 指派任何有空的工人（Node/Agent）來執行任務。
  • stages { ... }：流程總管，包含多個階段。
  • stage('Build')：具體階段，例如「備料」、「烹飪」或「打包」。
  • steps { ... }：實際動作，這裡寫具體的指令，如 sh (執行 Shell 指令) 或 echo (印出訊息)。 
pipeline {
    agent any
    // 關鍵：宣告使用的工具，名稱需對應 Jenkins 系統設定
    tools {
        maven 'Maven 3'
        jdk 'JDK 17'
    }
    stages {
        stage('Checkout') {
            steps {
                // 顯示拉取程式碼過程
                checkout scm
            }
        }
        stage('Build') {
            steps {
                echo 'Building...'
                // -B (Batch mode) 讓 Log 更簡潔，-DskipTests 跳過測試加速驗證
                sh 'mvn clean package -B -DskipTests'
            }
        }
        stage('Test') {
            steps {
                echo 'Testing...'
                sh 'mvn test'
            }
        }
    }
    // 建置後的動作：清理、通知與錯誤處理
    post {
        always {
            cleanWs() // 清理工作區，節省空間
        }
        success {
            echo '✅ 建置成功！'
            // 可選：發送成功通知到 Slack
            // slackSend(channel: '#devops', color: 'good', message: "✅ 建置成功: ${env.JOB_NAME} #${env.BUILD_NUMBER}")
        }
        failure {
            echo '❌ 建置失敗，請檢查 Log'
            // Email 通知範例（需安裝 Email Extension Plugin）
            emailext(
                subject: "❌ Jenkins 建置失敗: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
                body: """建置失敗，請檢查日誌：
                         Job: ${env.JOB_NAME}
                         Build: ${env.BUILD_NUMBER}
                         URL: ${env.BUILD_URL}""",
                to: 'dev-team@example.com'
            )
        }
        unstable {
            echo '⚠️ 建置不穩定，存在警告'
        }
        aborted {
            echo '🛑 建置已中止'
        }
    }
}

🔐 憑證安全管理

正確管理敏感資訊是 DevOps 安全的基石，避免密碼、Token 洩露於程式碼或日誌中。

❌ 絕對不要這樣做

// 🚫 寫死密碼或 Token（任何人都能看到！）
stage('Deploy') {
    steps {
        sh "curl -u admin:MyPassword123 https://api.example.com/deploy"
    }
}

// 🚫 把密碼放在環境變數中直接使用
stage('Build') {
    environment {
        DB_PASSWORD = 'SuperSecret123'  // 會出現在 Log 中！
    }
    steps {
        sh "echo ${DB_PASSWORD}"  // 密碼會被記錄
    }
}

✅ 正確做法：使用 withCredentials

stage('Deploy with Token') {
    steps {
        // 使用 Jenkins Credentials Store 中的憑證
        withCredentials([string(credentialsId: 'api-token', variable: 'API_TOKEN')]) {
            // Jenkins 會自動在 Log 中隱藏 TOKEN 內容
            sh '''
                curl -H "Authorization: Bearer ${API_TOKEN}" \
                     https://api.example.com/deploy
            '''
        }
    }
}

// 使用多種憑證類型
stage('Database Migration') {
    steps {
        withCredentials([
            usernamePassword(
                credentialsId: 'db-credentials',
                usernameVariable: 'DB_USER',
                passwordVariable: 'DB_PASS'
            )
        ]) {
            sh '''
                mysql -u ${DB_USER} -p${DB_PASS} < migration.sql
            '''
        }
    }
}

💡 關鍵優勢：
• 憑證集中管理於 Jenkins Credentials Store
• 自動遮蔽 Log 中的敏感資訊
• 支援憑證輪替而不需修改 Pipeline 程式碼
• 可整合外部 Secret 管理工具（如 HashiCorp Vault）

🔧 設定 Jenkins Credentials

  1. 進入 Manage Jenkins → Credentials
  2. 選擇對應的 Domain（通常是 Global）
  3. 點選「Add Credentials」
  4. 選擇憑證類型：
    • Secret text： API Token、密碼
    • Username with password： 資料庫帳密
    • SSH Username with private key： SSH 私鑰
    • Certificate： SSL/TLS 憑證
  5. 設定易識別的 ID（例如 github-api-token）
  6. 在 Pipeline 中透過 credentialsId 引用

🚀 自動化部署實戰流程

整合 GitHub、Maven 與 SSH，實現從程式碼提交到伺服器上線的全自動化流程。

01

設定 GitHub 權限與 Webhook

為了解決 Jenkins 拉取私有儲存庫程式碼的權限問題，需設定 SSH Keys 或 Personal Access Token (PAT)。
設定路徑：Manage Jenkins → Credentials → Add Credentials。 若需實現「Push 即建置」，需完成以下設定：
1. Jenkins Job 設定：

  • 進入 Job Configuration → Build Triggers
  • 勾選 「GitHub hook trigger for GITScm polling」
2. GitHub Webhook 設定：
  • Payload URL: http://<你的Jenkins IP>:8080/github-webhook/ ⚠️ 本地/內網環境需建立外部通道。傳統可用 ngrok/frp 等反向代理，或推薦使用更安全簡單的 Zero Trust 方案：使用 Tailscale 建立安全通道
  • Content type: application/json
  • 觸發事件: Just the push event

02

專案建置與測試

在 Pipeline 中定義 Maven 或 Gradle 指令。需要在 Jenkins 的「Global Tool Configuration」中預先設定好 JDK 與 Maven 的路徑，確保執行環境變數正確。
建議：將 Maven 相依套件快取 (m2 repo) 掛載至容器外部，加快建置速度。

03

部署至遠端伺服器 (使用 Jenkins Credentials/sshagent)

建置成功後，利用 SSH 將 Artifact 傳輸至目標伺服器。建議使用標準做法，將私鑰託管於 Jenkins。
關鍵步驟 (修正版)：

  • 在本機/實體主機產生金鑰： 使用 ssh-keygen -t rsa -f jenkins_key 生成一對金鑰。

    🔑 新手觀念： SSH Key 包含兩部分。
    1. 私鑰 (Private Key)：像是家裡的鑰匙，必須自己保管好 (交給 Jenkins 保管)。
    2. 公鑰 (Public Key)：像是鎖頭，可以複製很多個裝在不同伺服器上 (放在目標機器的 authorized_keys)。
    這樣拿著鑰匙 (Jenkins) 就能打開裝有鎖頭 (目標伺服器) 的門。

  • 設定目標伺服器： 將公鑰 (jenkins_key.pub) 內容複製到目標伺服器的 ~/.ssh/authorized_keys 檔案中。
  • Jenkins 憑證託管： 進入 Manage Jenkins → Credentials，新增類型為 "SSH Username with private key"，將私鑰 (jenkins_key) 內容貼入，並記下 ID (例如 remote-server-ssh)。
  • Pipeline 部署腳本： 使用 sshagent 套件進行連線： 
    stage('Deploy') {
    steps {
        sshagent(['remote-server-ssh']) {
            sh '''
                # 建立 .ssh 目錄並自動掃描主機指紋 (使用 -H 雜湊主機名)
                mkdir -p ~/.ssh
                ssh-keyscan -H <目標IP> >> ~/.ssh/known_hosts 2>/dev/null
                
                # 遠端部署
                ssh user@<目標IP> '
                    cd /var/www/app && 
                    echo "Deploying..." &&
                    ./deploy.sh
                '
            '''
        }
    }
}

    ⚠️ 避免使用 -o StrictHostKeyChecking=no，這會讓你暴露於中間人攻擊（MITM）風險中！

🛡️ 安全性與套件管理

維護 Jenkins 安全性與穩定性的關鍵策略。

📦 套件版本管理策略

1. 使用 LTS 版本

• Jenkins 提供長期支援版本（LTS），每 12 週發布一次
• 建議生產環境使用 LTS 版本而非每週更新版
• 下載位置：jenkins/jenkins:lts Docker image

2. 鎖定套件版本

• 避免設定「自動更新套件」
• 在測試環境先驗證新版本套件再部署到生產環境
• 使用 plugins.txt 或 Configuration as Code (JCasC) 管理套件版本

3. 定期檢查安全公告

• 訂閱 Jenkins Security Advisories
• 在「Manage Jenkins → System Information」查看套件版本
• 使用「Plugin Manager」查看可用更新與安全修補

🚨 最小化攻擊面

  • 移除不必要的套件： 定期檢視已安裝套件，移除未使用的套件
  • 限制 Master 節點建置： 設定「# of executors」為 0，所有建置都在 Agent 執行
  • 啟用 CSRF（Cross-Site Request Forgery）保護： Manage Jenkins → Configure Global Security → Prevent Cross Site Request Forgery exploits
  • 使用授權矩陣： 精細控制不同使用者/團隊的權限（Matrix-based security）
  • 定期備份： 備份 jenkins_home，包含所有設定與 Job 歷史

🔌 進階功能與最佳化

提升 Jenkins 使用體驗與效能的關鍵設定。

Blue Ocean 介面

Jenkins 的全新 UI 套件。提供視覺化的 Pipeline 流程圖，讓每個階段 (Stage) 的狀態、日誌與錯誤訊息一目瞭然，大幅提升了使用者體驗與除錯效率。

分散式建置 (Nodes)

為避免 Master 節點負載過重，可設定多個 Agent 節點。連接方式選擇建議：

連接方式 適用場景 優缺點
SSH 標準企業內網 Master 主動連線，簡單直接
JNLP 防火牆後的 Agent Agent 主動回連，安全性較高
Docker Cloud 動態擴展需求 隨用隨丟，資源不浪費

💡 小撇步： 確保 Master 與 Agent 的 Java 版本一致，且 Agent 節點上也需安裝 Git 與 Docker 才能執行對應的 Pipeline 指令。
