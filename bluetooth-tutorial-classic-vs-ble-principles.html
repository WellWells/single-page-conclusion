起源故事與符文原理

維京盟主與符文融合：
藍牙標誌的由來

1994 年，瑞典電信巨頭 Ericsson 的工程師極力尋找取代有線通訊的方法。1998 年，他們聯手 IBM、Intel、Nokia 與 Toshiba 成立了藍牙技術聯盟 (Bluetooth SIG)。

當時參與專案的 Intel 研發人員 Jim Kardach 建議將此技術命名為「Bluetooth」，源自 10 世紀統一丹麥與挪威的維京國王哈拉爾德·藍牙（Harald Bluetooth Gormsson）。

哈拉爾德名字的盧恩縮寫（Bindrune，結合符文）原理：

國王名字的首字母分別是代表 H 的 Hagall (ᚼ) 與代表 B 的 Bjarkan (ᛒ)。 將這兩個盧恩字母在垂直中軸線上完美重疊、穿插，即會神奇地誕生出全世界通用的 藍牙官方標誌。

藍牙規格演進里程碑

從 1.0 的替代電纜，到 6.0 的精準定位

藍牙並非一成不變。每一次主版本迭代，都解決了當時產業最急迫的痛點（傳輸速率、通訊功耗、定向安全以及空間測距）。以下為您深挖核心規格：

Bluetooth 1.x 1999 - 2003

定位：電纜替代 (Cable Replacement)

最初設計旨在取代印表機、滑鼠與鍵盤的連接線。隨著 Core Spec 迭代，應用層設定檔（Profiles）如 A2DP（立體聲音訊）雛形也隨之發展，但底層最大理論空中傳輸速率僅為 1 Mbps (Basic Rate)。

Bluetooth 2.x 2004 - 2007

定位：高速與配對安全強化 (EDR / SSP)

2.0 + EDR (Enhanced Data Rate)：核心規格層透過更換調變技術（π/4-DQPSK / 8DPSK）實現最高 3 Mbps 理論速率，為上層如 AVRCP 等應用層設定檔提供了更充裕的頻寬。
2.1 + SSP (Secure Simple Pairing)：引入安全簡易配對，透過數值比較等機制，在無需繁瑣輸入 PIN 碼的情境下建立防範中間人（MITM）攻擊的安全鏈結。

Bluetooth 3.0 2009

定位：檔案高速傳輸 (High Speed)

引進了 AMP (Alternative MAC/PHY) 技術。藍牙建立連線後，會協同並調配本地 Wi-Fi 射頻模組，以高達 24 Mbps 的速度傳輸大型檔案，完成後立刻切回低速藍牙省電。

Bluetooth 4.x 2010 - 2014

定位：BLE 革命與物聯網擴充 (完全不向下相容)

4.0 (BLE 誕生)：這是一個歷史分水嶺，推出了與傳統經典藍牙完全不相容的低功耗藍牙 (BLE)，專攻超低耗電感測器，自此藍牙正式分為雙軌制。
4.2 (GATT 隱私升級)：支援 DLE (Data Length Extension)，將單個封包的資料長度由 27 Bytes 擴大到 251 Bytes，傳輸效率大幅增加 2.5 倍。

Bluetooth 5.x 2016 - 2023

定位：極限速度、長距離與 LE Audio

5.0：提供 2 Mbps (高速模式) 與 Coded PHY (FEC前向錯誤更正碼) 長距離模式，讓距離大增 4 倍（可達 1 公里以上）。
5.2 (LE Audio)：引進 LC3 解碼與同步通道。
5.4 (PAwR)：支援加密廣播資料，使電子標籤與物聯網通訊更安全。

Bluetooth 6.0 最新發表 (2024/2026)

定位：Channel Sounding 高精定位基礎

Channel Sounding：結合「相位測距」(PBR) 與「飛行時間測距」(RTT)，提供公分級別的兩點間高精度一維直線距離（Range）量測，可作為多點三角定位系統的基礎，是數位車鑰與尋找配件的終極技術。
決策性廣播過濾 (Decision-Based Ad. Filtering)：優化主次廣播頻道跳轉決策，防止射頻晶片無謂喚醒，大幅節省功耗。

實體射頻與鏈結層對比

雙模藍牙的核心技術實體對比

傳統藍牙與 BLE 雖然同樣運作於無須授權的 2.4 GHz ISM 射頻頻段 (2.400 GHz 至 2.4835 GHz)，但其射頻實體層（PHY）、跳頻寬度、調變指數與工作週期邏輯是完全互不相容的。

Classic (BR / EDR) 持續高頻寬流傳輸

經典藍牙：高流量連續連結

經典藍牙設計的核心是「持續性的資料通道與立體聲音訊串流」。它採用較為密集的頻道配置以及高速的跳頻技術，用以維持極其穩定且低延遲的鏈結。

BLE (Smart) 間歇性突發低耗能

低功耗藍牙：極致省電與突發傳輸

BLE 拋棄了不間斷連線的包袱。它的核心實體原則是「極速醒來、在毫秒級時間內發送完短小封包（Burst Mode）、隨後瞬間重回零能耗睡眠」。

System Architecture

藍牙的「主機與控制器分離」：
Controller 與 Host

藍牙系統在設計上高度模組化。為了解決低端微控制器算力不足的問題，傳統架構將實體電路射頻與上層邏輯分層實作，中間使用 HCI (主機控制介面) 介接。

控制器層 (Controller Layer)

負責直接實體空中通訊與調變（GFSK等）。包含最底層的實體層(PHY)與連結層(Link Layer)，掌控著跳頻頻道選擇、重傳確認與射頻硬體。

主機層 (Host Layer)

提供具體的應用協定。包含邏輯鏈結控制與適應協定 (L2CAP)、安全管理器 (SMP)、屬性協定 (ATT) 與通用屬性設定檔 (GATT)。

低功耗藍牙核心邏輯

GATT 屬性管理與連線控制參數

在 BLE 中，資料暴露與連線交握具有極高效率。了解 GATT 的 Services / Characteristics 對應，以及 GAP 的廣播掃描流程，是開發穩定射頻應用的必經之路。

屬性協定 (GATT / ATT) 樹狀資料階層

Service UUID: 0x180D

心率服務 (Heart Rate Service)

邏輯性地封裝所有與心率相關的資料參數。

Characteristic UUID: 0x2A37

即時心率量測值

Read Notify
Descriptor UUID: 0x2902

用戶端特徵組態描述元 (CCCD)

用於讓中央用戶端（Central）訂閱或取消訂閱主動推送的通知。

Connection Interval

連線時間間隔

定義主從端進行資料同步與探測的頻率（通常介於 7.5 毫秒到 4.0 秒之間）。縮短此參數能提高傳輸速率（1.4 Mbps 峰值），但會犧牲功耗。

Peripheral Latency

周邊裝置延遲 (從機延遲)

允許周邊裝置在無最新資料時，暫時跳過（Skip）指定次數的連線間隔。避免頻繁發送空資料封包，進而極大延長鈕扣電池的電池壽命。

Supervision Timeout

監控連線超時

用以檢測鏈結中斷的最大允用超時。若在這個期限之內，雙方未能成功交握完成一次資料交換，則會強制宣告連線關閉並釋放通訊頻道。

以下展示在 ESP32 上實作「虛擬序列埠 (Serial UART)」功能的程式碼，幫助您用直覺、流暢的視覺感快速掌握底層 API 與邏輯差異。

C++ CODE IMPLEMENTATION
BluetoothSerial.h (經典藍牙通訊流範例) 
#include "BluetoothSerial.h"

// 宣告經典藍牙序列埠物件
BluetoothSerial SerialBT;

void setup() {
  Serial.begin(115200);
  
  // 初始化傳統藍牙射頻，設備名稱為「BT_Classic_Node」
  SerialBT.begin("BT_Classic_Node");
  Serial.println("經典藍牙開啟成功！可於外部設備進行 RF 配對與鏈結。");
}

void loop() {
  // 判斷本地主機硬體序列埠緩衝區是否有新輸入
  if (Serial.available()) {
    char inChar = (char)Serial.read();
    SerialBT.write(inChar); // 轉發字元給經典藍牙對端裝置
  }
  
  // 判斷藍牙接收緩衝區是否被填入新字節
  if (SerialBT.available()) {
    char btChar = (char)SerialBT.read();
    Serial.write(btChar); // 經由實體序列埠回傳給 PC 的 Serial Monitor
  }
  
  delay(10); // 避免頻率空轉
}
BLE UART (基於 Nordic NUS 規範) 實作範例 
#include <BLEDevice.h>
#include <BLEServer.h>
#include <BLEUtils.h>
#include <BLE2902.h>

// 採用標準 Nordic UART Service (NUS) 的 UUID
#define SERVICE_UUID           "6E400001-B5A3-F393-E0A9-E50E24DCCA9E"
#define CHARACTERISTIC_UUID_RX "6E400002-B5A3-F393-E0A9-E50E24DCCA9E"
#define CHARACTERISTIC_UUID_TX "6E400003-B5A3-F393-E0A9-E50E24DCCA9E"

BLEServer *pServer = NULL;
BLECharacteristic *pTxCharacteristic;
bool deviceConnected = false;

// 監聽連線狀態的回呼函式
class MyServerCallbacks: public BLEServerCallbacks {
    void onConnect(BLEServer* pServer) { deviceConnected = true; };
    void onDisconnect(BLEServer* pServer) { deviceConnected = false; }
};

// 處理來自手機/Client 端寫入(RX)事件的回呼函式
class MyCallbacks: public BLECharacteristicCallbacks {
    void onWrite(BLECharacteristic *pCharacteristic) {
      std::string rxValue = pCharacteristic->getValue();
      if (rxValue.length() > 0) {
        Serial.print("收到來自藍牙的訊息: ");
        for (int i = 0; i < rxValue.length(); i++) Serial.print(rxValue[i]);
        Serial.println();
      }
    }
};

void setup() {
  Serial.begin(115200);
  BLEDevice::init("BLE_UART_Node");

  // 建立伺服器與綁定連線事件
  pServer = BLEDevice::createServer();
  pServer->setCallbacks(new MyServerCallbacks());
  BLEService *pService = pServer->createService(SERVICE_UUID);

  // 建立 TX 特徵值 (Server -> Client，使用 Notify)
  pTxCharacteristic = pService->createCharacteristic(
                    CHARACTERISTIC_UUID_TX,
                    BLECharacteristic::PROPERTY_NOTIFY
                  );
  pTxCharacteristic->addDescriptor(new BLE2902()); // CCCD 描述元，用於通知訂閱

  // 建立 RX 特徵值 (Client -> Server，使用 Write)
  BLECharacteristic *pRxCharacteristic = pService->createCharacteristic(
                       CHARACTERISTIC_UUID_RX,
                       BLECharacteristic::PROPERTY_WRITE
                     );
  pRxCharacteristic->setCallbacks(new MyCallbacks());

  pService->start();
  pServer->getAdvertising()->start();
  Serial.println("BLE UART 啟動完成，等待手機端連線...");
}

void loop() {
  // 若雙方已連線且電腦實體序列埠有新字元輸入，則透過 Notify(TX) 傳給手機
  if (deviceConnected && Serial.available()) {
    char inChar = (char)Serial.read();
    uint8_t txValue[1] = { (uint8_t)inChar };
    
    pTxCharacteristic->setValue(txValue, 1);
    pTxCharacteristic->notify(); // 類似 SerialBT.write() 的動作
    delay(10);
  }
}
