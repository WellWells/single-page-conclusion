Shellcode 簡介

Shellcode 是一種機器碼（Machine Code），當被注入到目標處理序（Process）的記憶體中時，會執行特定的操作。它通常作為漏洞利用（Exploits）中的有效載荷（Payload），執行諸如開啟命令列 Shell、建立網路連線或執行任意指令等動作。

Shellcode 基本概念

Shellcode 執行模式

Shellcode 注入通常遵循「配置-寫入-執行」（Allocate-Write-Execute）模式：

  1. 配置大小與 Shellcode 相符的 RWX（可讀可寫可執行）記憶體。
  2. 將 Shellcode 寫入配置的記憶體中。
  3. 執行已配置記憶體中的內容。

OPSEC 警告

應避免直接使用 PAGE_EXECUTE_READWRITE 進行配置，因為這很容易被標記為可疑。較佳的做法是：

  1. 使用 PAGE_READWRITE 進行配置。
  2. 寫入有效載荷（Payload）。
  3. 將記憶體保護屬性更改為可執行（Executable）。

Windows Shellcode 機制

Windows Shellcode 的 API 解析流程

識別 kernel32.dll 與解析函數的完整過程：

  • 取得 PEB->Ldr.InMemoryOrderModuleList。
  • 迭代遍歷 InMemoryOrderModuleList；在 Win10/11+ 順序通常為：exe 本身 → ntdll.dll → kernel32.dll。透過名稱雜湊比對定位。
  • 解析匯出位址表（Export Address Table, EAT）。
  • 迴圈遍歷 EAT 尋找 GetProcAddress 的名稱。
  • 取得函數序號（Ordinal）後取得位址（獲得 GetProcAddress 權限）。
  • 在解析 LoadLibraryA 前確保 GetProcAddress 可用；兩者可在同一列舉迴圈獲取。
  • 使用 GetProcAddress 解析 LoadLibraryA 並載入 WS2_32.dll。
  • 取得並呼叫 WSAStartup。
  • 查詢 WSASocketA 建立 Socket。
  • 查詢 WSAConnect 連線到該 Socket。
  • 再次從 kernel32.dll 找到 CreateProcessA。
  • 將 cmd.exe 推入堆疊，填充 STARTUPINFOA 並呼叫 CreateProcessA。
  • 本地監聽：nc -nvv -p 443 -l。

Shellcode 載入器 (Loaders)

基本模式

C / C++
char *shellcode = "\xAA\xBB...";
char *dest = VirtualAlloc(NULL, 0x1234, 0x3000, p_RW);
memcpy(dest, shellcode, 0x1234);
VirtualProtect(dest, 0x1234, p_RX, &result);

認識 Shellcode 載入器

載入器負責將 Shellcode 作為輸入，驗證環境（Anti-Sandbox），解密並安全注入。它可能存在於處理序樹或模組列表中。

Rust (安全但體積大) Nim Go (特徵明顯) Zig (體積小/無依賴)

1. 配置階段 (Allocation)

  • 目標：新記憶體區域、Code caves、Stack/Heap。
  • 技術：VirtualAllocEx, NtAllocateVirtualMemory, ZwCreateSection。
  • 最佳實踐：配置 RW → 寫入 → 切換為 RX (避免直接 RWX)。

2. 寫入階段 (Write)

  • 技術：WriteProcessMemory, NtWriteVirtualMemory, memcpy。
  • 規避策略：
    • - 前置無用操作碼 (Dummy opcodes)
    • - 分塊 (Chunks) 隨機順序寫入
    • - 加入寫入延遲

3. 執行階段 (Execute)

最脆弱的步驟，EDR 重點監控。

  • CreateRemoteThread
  • NtSetContextThread / NtQueueApcThreadEx
  • API Hooks / Trampolines

跨平台考量

Windows on ARM64

Snapdragon X Elite / Surface Pro 10。Syscalls 使用 SVC 0 與 ntdll!KiServiceTableArm64。注意 PAC (指標驗證) 簽署 LR 值，避免 Stack pivots。

Linux 6.9+

引入 BPF Tokens 與共享 BPF Arena (可執行)。將 Shellcode 藏在 Arena map 中，透過 bpf_prog_run_pin_on_cpu 執行。

macOS SSV

系統卷密封。持久化需建立 Snapshot、注入並重簽 (kmutil)。使用者空間則利用 Launch Agents 或 dylib 劫持。

儲存、隱藏與規避技術

儲存選項

  • 寫死 (Hardcoded)：存於 RW/RO 區段，需重新編譯。
  • PE 資源：RCDATA Blobs (AV 重點掃描)。
  • 額外 PE 區段：倒數第 N 個 R+X 區段。
  • 網路獲取：如 CSharpShooter。
  • 證書表 (Certificate Table) ★推薦：填充 PE 簽章區塊，保持主程式簽章完整 (如 Teams.exe)，僅破壞依賴 DLL (如 ffmpeg.dll)。
  • 合法 PE 後門：植入 Shellcode 與 Loader。
保護方法： 壓縮 (LZMA) + 加密 (XOR32/RC4/AES)

規避技術 (Evasion)

漸進式規避策略

  1. 基本執行
  2. 加密/混淆
  3. 直接系統呼叫 (Direct Syscalls)
  4. 遠端程序注入 (Remote Injection)

本地 vs 遠端注入

遠端注入面臨 CFG, CIG, ETW 監控。偵測點包含 ETW Ti, EDR Hooks, 以及回溯執行緒呼叫堆疊 (Call Stack)。

進階實作：DripLoader 技術

  • 使用 NO_ACCESS 保留大塊記憶體。
  • 在保留池中配置分散的 4KB Read+Write 區塊。
  • 分塊隨機順序寫入 Shellcode。
  • 重新保護為 Read+Exec。
