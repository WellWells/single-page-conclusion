軟體開發的焦慮與反思 🤔

破除不必要的複雜性，回歸工程的樸素本質。

🧐 核心觀點摘要

許多軟體開發的困難與焦慮，源自於開發者過度思考與人為的複雜化。隨著技術理念的演進與硬體條件的升級，開發工作本應越來越容易。本文探討幾種常見的「自造焦慮」現象，並倡導「大道至簡」的解決方案。

😓 陷阱一：過度開發

開發者時常想像出不存在的需求，並投入大量精力去實現，以顯得「經驗老到」。

案例：為一個僅限國內市場的產品，強行導入完整的 Internationalization (i18n) 國際化框架，導致後續維護（如詞典同步、版本對應、快取問題）工作量劇增，而該功能始終未被使用。

另例：使用 Hadoop 處理僅有數 GB、完全可載入記憶體的資料。這種「高射炮打蚊子」的做法，不僅浪費數月的開發部署工時，且效能遠不如單純的記憶體內運算。

📜 陷阱二：無效的規範

在沒有統一規範時，團隊會為無意義的細節（如大括號位置、命名風格）爭論不休，浪費寶貴時間。

解決方案：應在專案初期就訂立統一規則，並整合到 Continuous Integration / Continuous Deployment (CI/CD) 流程中，透過工具自動化執行，消除爭議。

API 混亂：另一類混亂源自 Application Programming Interface (API) 的設計。應遵循成熟標準（如 Representational State Transfer (REST)ful），而非自作聰明，製造難以合作的工具。

😵 陷阱三：反人類的規範

以 "Clean Code" 為代表的某些規範，有時會陷入「為美觀而美觀」的盲點。

程式碼是工具，不是抽象藝術。高級程式語言的發明，是為了讓人更容易閱讀、撰寫與合作。

Unix 哲學強調「接地氣、不要花裡胡哨」（例如：文本是最好的 API 介面）。任何規則如果不是為了讓人的工作變得更容易，而是源於虛無飄渺的理由，那它本身的方向就是錯誤的。

📦 陷阱四：技術堆疊越大越好？

技術堆疊的規模與「技術債」成正比。堆疊技術並不能獲得上級賞識，交付功能才是。

例外：當一項新技術能 *取代* 多項舊技術時，它是有價值的。例如 Kubernetes 簡化了 DevOps 流程，從而 *縮小* 了整體的技術堆疊。

範例：Postgres (PostgreSQL) 不僅是 Structured Query Language (SQL) 資料庫，還能高效處理 JavaScript Object Notation (JSON)、圖形（取代 Neo4j）和 Key-Value（取代 Redis）資料。善用單一成熟技術，能大幅減少技術堆疊。

🎡 陷阱五：別再造輪子

為目前專案需求造一個「通用」輪子，是常見的陷阱。

開發者低估了維護通用工具的難度：它實質上已成為一個獨立產品，需要處理完整的生命週期、向後相容性、並應對未來所有新需求。

這導致絕大部分的輪子，在能被大規模應用之前，就因為維護難度過高而被開發者「棄坑」，成為技術負債。

🏆 陷阱六：最好的方案...是最樸素的

當資料庫裝不下時，首要考量的不該是分布式架構或集群——這類牽扯到 ACID 特性的方案，其複雜度是「圖靈獎級別」的。

最好的方案：拆分 (Sharding)。

在邏輯上無瓜葛的資料（如不同客戶）就分開存放；或將超大的表單獨移至一個資料庫。

美國的 Pinterest 早期僅靠 3 位程式員，拋棄複雜集群，將 MySQL 拆分為 88 個，便應付了千萬級別的月活躍使用者。

再如：使用 Excel 表格取代一個複雜的 Java Web Content Management System (CMS)，反而獲得使用者好評。

🌟 結論：大道至簡

軟體開發作為一種工程實踐，應回歸問題的本質。刻意追求複雜架構、堆疊時髦技術、或陷入虛無的規範之爭，只會徒增焦慮。

一個簡單、樸素、能快速解決問題的方案，往往才是最高效、最穩健的選擇。
