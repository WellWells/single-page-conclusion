關鍵發現 Summary

未授權的完整揭露

駭客 "Chaotic Eclipse" 於 GitHub 發布全功能 PoC。微軟目前僅回應「支援協調揭露」，尚無 CVE 編號或修補程式。

合法功能的致命連鎖

漏洞並非源於程式碼錯誤，而是 VSS 陰影複製、Cloud Files API 與 Opportunistic Locks (oplocks) 三種合法功能的惡意連鎖。

影響範圍極廣

在低權限使用者環境下，可在 1 分鐘內提升至 SYSTEM 權限。已在完全修補的 Windows 10/11 上驗證成功。

特徵碼繞過

儘管 Defender 已更新靜態偵測特徵碼，但由於是架構性缺陷，攻擊者只需重新編譯或微調參數即可輕易繞過。

漏洞究竟是什麼？

BlueHammer 是一種「設計缺陷」與「競爭條件 (Race Condition)」的結合。

VSS 濫用

Defender 在更新與清理時會建立暫存的磁碟陰影複製快照，這成為一個存取系統核心檔案（如 SAM Hive 檔）的後門。

Cloud Files API

利用雲端檔案回呼機制，攻擊者可以獲得精確的控制權，決定何時「回應」Defender 的讀取請求。

機會鎖定 Oplocks

這是「暫停按鈕」。透過 Oplocks 將 Defender 行程凍結在特定、可預期的時刻，讓快照持續掛載供讀取。

技術分析：攻擊鏈詳解

「攻擊並非源於 Bug，而是 Defender 可以被脅迫在錯誤的時間暫停自己，讓特權資源門戶大開。」

01

偵測 Defender 特徵碼更新

PoC 首先透過 Windows Update COM 介面查詢擱置的更新。漏洞觸發的先決條件是必須有特徵碼更新正在進行中，否則 Defender 不會啟動 VSS 建立程序。

// 偽程式碼：查詢 Defender 定義更新
IUpdateSearcher* searcher = ...;
ISearchResult* results = searcher->Search("Type='Software' AND CategoryID='...'");
02

誘騙 Defender 啟動掃描

程式會建立一個暫存目錄並丟入 EICAR 測試病毒檔。同時對特定 DLL 掛上 Oplock 作為觸發器，當 Defender 觸碰該檔案時，代表 VSS 行程已啟動。

03

建立「陷阱」同步根

利用 CfRegisterSyncRoot 註冊雲端檔案同步目錄。當 Defender (WinDefend 行程) 嘗試枚舉此目錄時，系統會調用攻擊者掌控的回呼函式。

04

凍結 Windows Defender

關鍵點：當回呼確認是 Defender 在請求檔案時，它會拒絕返回並持有 Oplock。Defender 會因為等待回應而陷入無限期封鎖。

此時，Defender 剛建立的 VSS 快照仍處於掛載狀態且可供讀取。

05

竊取登錄檔 Hive (Registry Hives)

趁 Defender 被凍結，攻擊者直接從陰影複製路徑讀取通常被系統鎖定的檔案：

  • \\Device\\HarddiskVolumeShadowCopyX\\Windows\\System32\\Config\\SAM
  • \\Device\\HarddiskVolumeShadowCopyX\\Windows\\System32\\Config\\SYSTEM
06

密碼還原與權限提升

使用標準技術解密 LSA secrets 與 NTLM hash。隨後透過 samlib.dll 強行更改本機管理員密碼並登入，奪取 SYSTEM 權杖。

07

自我清理與潛伏

在產生 SYSTEM Shell 後，PoC 會立即將管理員密碼還原為原始的 Hash，從使用者的角度看密碼毫無變化，極難察覺。

DEEP DIVE ANALYSIS

專案核心架構與運作原理剖析

一、 專案核心邏輯與流程

這個專案的核心邏輯是不透過微軟官方提供的 Windows Defender API，而是直接建構惡意的 RPC 客戶端，向高權限的 Defender 服務發送特製指令。整套系統的運作流程推斷如下：

1. RPC 綁定 (Binding)

透過 windefend_c.c 建立與 Defender ALPC (Advanced Local Procedure Call) 通訊埠的連線。

2. 環境建構 (Staging)

利用 offreg.h 建立特製的離線登錄檔 (Offline Registry Hive)，或利用 ktmw32.h 建立 NTFS 交易環境。

3. 觸發漏洞 (Trigger)

呼叫 windefend_h.h 中定義的非公開 RPC 函式（如要求載入設定、掃描隔離檔），誘使高權限的 MsMpEng.exe 處理攻擊者準備好的惡意檔案。

4. 利用與清理 (Exploit & Cleanup)

達成寫入系統等目的後，FunnyApp.cpp 尾段會執行清理工作，抹除暫存檔案與目錄痕跡，達到無痕退出。

二、 檔案實作細節深度拆解

1. FunnyApp.cpp 攻擊核心與控制邏輯

這是整個 PoC 的進入點。透過詳細閱讀原始碼可發現其標頭檔暴露了許多意圖：

  • <Shlwapi.h>, <cfapi.h>, <aclapi.h>：用於進階檔案路徑操作、雲端檔案 API (Cloud Files) 及存取控制清單 (ACL) 修改。這證實了漏洞利用高度依賴檔案權限的競爭條件 (Race Condition)。
  • 核心關鍵 <ktmw32.h>：核心交易管理員 (KTM)。在許多 LPE 漏洞中，攻擊者利用 TxF 將檔案操作包裝在交易中，配合符號連結 (Symlink) 欺騙防毒軟體或系統服務寫入錯誤的對象。
  • <wuapi.h>：Windows Update API，用於與 Defender 的更新機制連動以觸發漏洞。

2. windefend_h.h, _c.c, _s.c 逆向工程 RPC 介面

這三個檔案是透過微軟 MIDL 編譯器自動生成。攻擊者首先逆向分析了 MsMpEng.exe，提取出其 RPC 介面定義檔 (.idl)，然後編譯出這組 C 語言的 Stub。

  • UUID： 8A885D04-1CEB-11C9-9FE8-08002B104860。這是 Windows Defender 核心 RPC 介面的唯一識別碼。
  • windefend_h.h (標頭檔)： 定義了所有可呼叫的遠端函式原型。微軟並未公開這些函式名稱，因此逆向工程師會以 ProcXXX 命名，並依參數類型推測其功能。
  • windefend_c.c (Client Stub)： 編譯時連結的檔案，負責將函式呼叫打包 (Marshaling) 成網路/ALPC 封包並發送給 Defender 服務。
  • windefend_s.c (Server Stub)： 伺服器解包邏輯。在此 PoC 中不需編譯，除非要架設假 Defender 服務攔截通訊。

3. offreg.h 脫機登錄檔操作

微軟官方的 Offline 登錄檔 Library。允許程式在不依賴 Windows 登錄檔管理員的情況下，直接讀寫登錄檔 Hive 檔案 (如 NTUSER.DAT)。

用途推斷：Defender 的某些掃描或還原機制可能相依於登錄檔設定。攻擊者使用 ORCreateHive、ORSetValue 等函式在磁碟構造充滿惡意指令/路徑的「假」登錄檔。再透過 RPC 強迫 Defender 載入這份假配置，從而觸發記憶體損毀或邏輯錯誤。

4. FunnyApp.sln, FunnyApp.rc, resource.h 專案建置與資源

標準 Visual Studio 專案結構。.sln 定義支援 x86/x64 的編譯設定；.rc 與 resource.h 定義應用資源 (如版本資訊、Icon 等)，在此 PoC 中屬於輔助性質。

偵測 VSS 枚舉行為

監控非系統行程對 NtQueryDirectoryObject 的調用，特別是目標為 HarddiskVolumeShadowCopy* 的行為。這在一般使用者環境中幾乎沒有合法用途。
