為什麼我們需要關心 DNS 安全？

每當您訪問一個網站，您的設備都會發出一個 DNS (域名系統) 查詢，將人類可讀的域名 (如 www.google.com) 轉換為機器可讀的 IP 地址。然而，傳統的 DNS 協議是以明文形式傳輸的，這帶來了嚴重的隱私和安全風險。本頁將引導您了解主要的威脅，並探討現有的和未來的解決方案。

兩大核心威脅

👁️ 上網行為監控

由於傳統 DNS (使用 UDP/53 或 TCP/53 連接埠) 查詢是未加密的，您訪問的每一個網站域名都可能被網路路徑上的任何人窺探，包括您的網路營運商 (ISP) 或公司雇主。這些記錄可以被長期保存，用於分析您的上網習慣。

解決方案：功能比較與限制

為了解決上述威脅，社群開發了多種技術。沒有任何一種技術是完美的，它們各自有其適用場景、限制與被封鎖的風險。下表將進行詳細比較：

加密DNS: DoH (DNS over HTTPS)

將 DNS 查詢封裝在 HTTPS 流量中，提供傳輸加密。

  • ✔️ 能防範: 監控、用戶端↔解析器路徑上的竄改/劫持。
  • ❌ 限制: 無法防止惡意解析器、未簽名上游偽造、政策封鎖。
  • 📡 連接埠: TCP/443, UDP/443。與一般網頁流量共用，抗封鎖性最強。

加密DNS: DoT (DNS over TLS)

使用 TLS 直接加密 DNS 流量，提供傳輸加密。

  • ✔️ 能防範: 監控、用戶端↔解析器路徑上的竄改/劫持。
  • ❌ 限制: 無法防止惡意解析器、未簽名上游偽造、政策封鎖。
  • 📡 連接埠: TCP/853。因使用專用埠，較易被防火牆直接封鎖。

加密DNS: DoQ (DNS over QUIC)

使用 QUIC 協議傳輸 DNS，提供傳輸加密與更佳效能。

  • ✔️ 能防範: 監控、用戶端↔解析器路徑上的竄改/劫持。
  • ❌ 限制: 無法防止惡意解析器、未簽名上游偽造、政策封鎖。
  • 📡 連接埠: UDP/853。與 DoT 同樣使用專用埠，易被封鎖。

DNSSEC (DNS Security Extensions)

透過數位簽章提供來源驗證與完整性，確保 DNS 回應未被竄改。

  • ✔️ 能防範: 快取污染、DNS 回應偽造。
  • ❌ 限制: 不提供加密、僅對已簽名網域有效、需解析器啟用驗證功能。
  • 📡 連接埠: n/a (屬於 DNS 協議擴展)。

HTTPS / TLS

不直接修復 DNS，但其憑證驗證機制提供了重要的間接防護。

  • ✔️ 能防範: 阻止瀏覽器連線到憑證錯誤的網站，降低釣魚風險。
  • ❌ 限制: 無法防禦使用有效憑證的釣魚域名（如 aple.com 冒充 apple.com）。
  • 📡 連接埠: TCP/443。

務實的組合方案與未來展望

沒有單一的「終極解決方案」，但我們可以組合出現階段最務實的防護策略：加密 DNS (優先考慮 DoH 以抗封鎖) + DNSSEC 驗證。

這個組合能同時兼顧傳輸過程的機密性與資料來源的真實性。然而，我們仍需認知其限制：它無法完全消除解析伺服器本身的可見性（伺服器管理者仍知道你的查詢），也無法對抗基於 IP 位址或國家政策層級的連線封鎖。未來，搭配 ECH (加密用戶端問候) 等新技術，將能進一步收斂隱私洩漏的攻擊面。
