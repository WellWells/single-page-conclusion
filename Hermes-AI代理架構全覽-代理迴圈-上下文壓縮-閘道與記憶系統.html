🤖 AI 代理・系統架構・LLM 應用

把 Hermes 拆開來看，
一套會「越用越懂你」的 AI 代理長什麼樣

這是一趟 Hermes 代理程式的高層次導覽。看完之後，你不只更知道怎麼用它，也能理解：如果哪天想自己打造一隻類似的代理，背後那些迴圈、上下文、閘道與記憶，到底是怎麼接起來的。

📌 三分鐘先抓重點

  • ▸整套系統的核心，是一個極為簡單的代理迴圈；你可以透過命令列、閘道或 API 三種方式連到它。
  • ▸它的「個性」與「記性」其實只是幾個 Markdown 檔：soul.md（人格）、user.md（你是誰）、memory.md（學到的雜項事實）。
  • ▸它不靠斷詞器算 token，而是用「字元數除以 4」估算上下文長度，超過門檻就觸發壓縮，把舊訊息摘要掉。
  • ▸閘道（Gateway）是讓它紅起來的關鍵——它把代理接上 Telegram、Slack、電子郵件，並負責從 SQLite 重建整段對話。
  • ▸記憶分三層：Markdown 檔、SQLite 全文逐字稿、可選的外部記憶；排程工作則靠一個每分鐘跳一次的迴圈在驅動。

🐦 架構鳥瞰圖：其實組件少得驚人

先從最高處俯瞰整套架構。第一眼會發現：它的組件其實非常少。

最中心的，是一個 AI 代理核心（agent core），也就是它的代理迴圈。而要和這個核心溝通，有三條路：

連線方式使用情境
命令列介面（Command-Line Interface，CLI）直接在終端機輸入 hermes，當下就和代理對話。
閘道（Gateway）一個持續執行的系統，讓你透過 Telegram、電子郵件、Slack 等管道找到你的代理。
應用程式介面（Application Programming Interface，API）以程式呼叫的方式接上代理，串進其他服務。

這三條路最後都通往同一個代理核心。而圍繞著這個核心，還掛著一堆服務。安裝 Hermes 時，很多功能是開箱即用、預先裝好的：一整套工具（tools）、許多技能（skills），以及記憶（memory）。

💡 記憶從一開始就分成兩半

一半是外部記憶（external memory），由 mem0、SuperMemory 這類外部供應商提供（稍後詳談）；另一半是內部記憶（internal memory），基本上就是每段對話的逐字稿——你和代理的每次交談，都會以工作階段逐字稿的形式存進它的內部記憶。

除此之外，還有幾個可以修改的檔案，例如 soul.md 與 user.md，分別告訴代理「它自己的個性」以及「現在正在用它的人是誰」。整套系統的主要元素就這些，看起來真的很簡單——但接下來會一塊一塊講細。

🔁 代理迴圈：一個簡單到不行的循環

Hermes 內部的代理迴圈非常、非常單純，和其他走極簡路線的代理（例如 Pi、OpenCode）相當類似。它的運作節奏，會在每次使用者送出訊息時重複一遍：

  1. 使用者送出訊息。
  2. Hermes 開始組裝上下文：把它所有的內部記憶、為這次準備的提示，全部從「盒子」裡拿出來拼起來。
  3. 整段上下文與訊息歷史，連同系統提示，一起送進大型語言模型（Large Language Model，LLM）。系統提示裡就包含了 soul.md 與 user.md。
  4. LLM 可以決定呼叫工具。若它要呼叫，就執行該工具，再把結果回饋給 LLM。
  5. 只要 LLM 還想繼續用工具，第 4 步就一直重複（例如先網頁搜尋、再寫檔、再讀檔或更新檔案系統裡的某些檔案）。
  6. 所有需要的工具都呼叫完，LLM 給出最終回應。
  7. 記憶更新：回應送出後，再分析這次對話，看看有沒有值得記下、值得寫進記憶的東西。

最後這一步才是整個迴圈真正有意思的地方。它讓代理能在一次次互動中學習——下次你問類似的問題，它就已經學會了。這也正是讓 Hermes 成為一隻「越用越上手、越用越懂你」的代理的根本原因。

就這樣。代理迴圈本身非常直白：每來一則訊息，就把這套流程跑一遍。真正比較有趣的，是它周邊的那些部分。

🧩 上下文：幾個 Markdown 檔撐起一切

迴圈裡的「組裝上下文」值得單獨拉出來講，因為打造任何代理時，這一段都極為關鍵。而 Hermes 的上下文同樣走極簡路線——它主要就由幾個 Markdown 檔構成。

🎭 soul.md — 代理的人格

這是送進 LLM 的上下文中第一個、也最重要的部分。你在這裡寫下希望代理用什麼語氣說話、被什麼驅動、目標是什麼、該做什麼，以及它對你採取什麼態度。簡單說，任何你希望它表現出來的樣子，都寫在這裡。如果你看過 Claude 那種寫得很漂亮的巨大系統提示，那正是你想在 soul.md 裡為自己量身寫的東西。

💡 預設情況下，soul.md 是空的

剛裝好 Hermes 時，這個檔通常完全空白。如果你不自己設定、也不請 Hermes 幫你寫，它就會被一段預設系統提示取代——大致是告訴代理「你是 Hermes，一個永遠開機的虛擬助理」之類。但你大概會想擁有自己的 soul.md，這樣才能真正把代理調成屬於你的樣子。

🙋 user.md — 你是誰

它放在記憶目錄裡。和 soul.md 不同，Hermes 會在得知你的事情時自動更新它。假設你在對話中介紹自己：「對了，我是一名軟體工程師，正在做某個專案」或「我是市場分析師，負責某項工作」——一旦代理確認那是關於你的資訊，就會把它存進這裡。換句話說，這是從對話中累積出來的使用者輪廓。

🗒️ memory.md — 隨手記下的雜項事實

同樣在記憶目錄裡，但這個檔比較像是對任意事實的隨手筆記。它不一定和「你」有關，而可能是：某個工具怎麼用、某套工作流程怎麼跑、在對話中學到的有趣或有用的東西。代理會自動更新它（你也可以自己改、或要它更新）。只要它從你身上學到覺得有用的東西，並符合 soul.md 設定的目標，就會寫進 memory.md，留給未來的對話用。

除了這幾個 Markdown 檔，上下文裡通常還會出現別的東西：過去對話的相關資訊（過往工作階段的紀錄）；如果你設定了外部記憶，與此刻話題相關的歷史摘要也會被帶進來（沒設定就不會出現）；再來是技能說明與工具說明；最後，當然是你最近和 Hermes 往來的那些訊息。

最後這段訊息可以是整段對話；但一旦對話長到超過你代理上下文視窗的某個門檻，Hermes 的做法是把對話的摘要再做摘要。這正好帶到下一個主題：上下文壓縮。

🗜️ 上下文壓縮：什麼時候壓、又怎麼算

和 LLM 對話時，會有一個固定的上下文視窗，容量大約落在 25 萬到 100 萬個 token 之間。Hermes 的做法是：設定時就會問你，要在什麼時候觸發訊息歷史的壓縮。預設是用到上下文的 50%。

一旦越過這個門檻，Hermes 就會呼叫壓縮函式：把先前的訊息全部摘要成一段摘要，附回上下文，並用這段摘要取代原本那些訊息。這個門檻當然可以自訂——如果你用的是較小的模型，或上下文視窗不大，設定時它會問你要維持預設的 50%，還是改成 70%、80%。後者往往是更聰明的選擇。

⏱️ 兩個檢查時機

它會在兩個時刻檢查是否該壓縮：

時機說明
每一輪之前在呼叫 LLM、也就是每個回合開始前，先檢查一次。你每送一則訊息，它都會在那次迭代中檢查上下文。
發生錯誤時每當 LLM 回傳「上下文視窗超量」的錯誤，它就會回頭檢查上下文並摘要。

🧮 它怎麼知道「用了多少」

問題來了：要怎麼量目前的上下文有多長？這裡有兩種做法。

第一則訊息時——此時你還沒把任何東西送給 LLM，只是剛把整段上下文組好，手上還沒有任何 token 數量的資料。你當然可以動用斷詞器去精算，但 Hermes 用了更直白的辦法：把總字元數除以 4，當作上下文長度的近似值。只要這個估算超過 50%（或你設定的門檻），就觸發摘要壓縮。

第一則訊息之後——有更好的辦法。每當你送訊息、LLM 回你一段回應時，回應裡會帶一個用量（usage）參數：依供應商不同，有時含輸入、有時含輸出 token，有時直接給一個 usage 欄位，把它加總即可。這個數字更準，因為它用的正是你所用模型的同一套斷詞器。

💡 為什麼不一律用斷詞器精算？

在組裝階段就跑一次完整斷詞器，雖然辦得到，但太貴了。「字元數除以 4」已經夠準、又幾乎不花成本；等真的送出後，再用回應裡免費附帶的 usage 來校正。一種務實到位的取捨。

📝 壓縮提示裡，到底要 LLM 摘要出哪些東西

負責壓縮的提示，放在 context_compressor.py 這個檔裡，大約落在第 1400 行附近（每次重構都會挪動，但就在那一帶）。它的核心，是請 LLM 把整段對話摘要成好幾個固定區塊，例如：

摘要區塊內容
整體目標這件事最終想達成什麼。
限制條件過程中有哪些約束。
已完成動作目前為止做了哪些事。
目前狀態與進度活躍狀態、歷史進度、目前卡在哪、正在做什麼。
關鍵決策與已解問題做過哪些關鍵決定、哪些疑問已經有了答案。
相關檔案與關鍵脈絡牽涉到的檔案、不可遺漏的重要背景。
先前摘要與接續回合過往摘要、要納入的後續回合、待摘要的回合等。

可以看出，它要摘要的東西相當多——明顯比 Pi 那種更精簡的版本「肉」得多。先前談 Pi 架構時看過它的壓縮提示，那是個小得多、極簡得多的版本；而 Hermes 這分內容豐富許多，必然能給代理更完整的上下文，讓它更清楚現在到底發生了什麼事。

🚪 閘道（Gateway）：讓代理住進 Telegram、Slack 與你的信箱

閘道，就是讓你能透過各種通訊平台（Telegram、WhatsApp、電子郵件、簡訊、Slack 等）和代理對話的那一層。它在 Hermes 裡不見得是技術上最難的部分，卻很可能是讓 Hermes 變得這麼主流、這麼受歡迎的關鍵。

它的工作，是傾聽各路進來的訊息（前提是該閘道已設定好），把訊息整理成正確格式，組好上下文與整段對話後，再送給 AI 代理。

🔄 一個持續輪詢的非同步迴圈

運作上，它會起一個 asyncio 的非同步迴圈，持續執行、不斷去輪詢或拉取各個閘道。而每一種閘道，被拉取的方式都不一樣：

  • ▸有些用 Webhook。
  • ▸有些是一個每秒跑一次的小迴圈去呼叫對方系統。例如 Telegram，就可以用一個每秒輪詢一次 API、看機器人有沒有新訊息的小迴圈。
  • ▸有些則走 WebSocket。

正因如此，閘道在整套系統裡其實佔了不小的份量：每一個第三方整合，都得單獨設定，沒有哪個閘道是預設就能通吃一切的。設定流程大致是執行 hermes setup gateway，挑 Telegram，建立你的機器人 ID，再建立允許和代理溝通的使用者 ID。

🧱 它得從頭重建整段對話

閘道不只是收訊息而已。在命令列裡對話時，介面本身會記著整段對話、把完整歷史送回去；但走閘道時不是這樣——當閘道從 Telegram 收到訊息，它收到的只有那一則，不是整段對話。所以閘道得自己把訊息歷史與上下文重新組起來。也因此，迴圈裡的「組裝上下文」這一步，放到閘道的脈絡下才真正說得通：它每收到一則訊息，都得從零重建一次。

💡 訊息歷史怎麼找回來？靠工作階段 ID

識別碼的第一段是閘道名稱（例如 telegram），接著是 Telegram 回傳的工作階段 ID，再加上其他幾個 ID 拼成完整鍵值。所有對話的每一則訊息，都存在本機的 SQLite 資料庫裡。收到新訊息時，閘道就拿 telegram 前綴接上對應的工作階段 ID，去 SQLite 撈出該對話的完整歷史，附進上下文，再送給代理。

🚦 工作階段管理員：中斷、引導，還是排隊

還有個別忘了提的角色——閘道裡有個類似工作階段管理員的東西，負責決定「哪些訊息、在什麼時機」真正送到 LLM。舉例來說，當代理正在處理你上一則訊息時，你又緊接著送一則進來，它就得判斷該怎麼辦，而這取決於你怎麼送：

你怎麼送它怎麼做
/interrupt直接中斷代理目前的工作。
/steer引導、調整它正在做的事。
直接送（不加指令）排入佇列，等它忙完再處理。

所以閘道是個相當好玩、也很實用的部分——它就是把 Hermes 接上所有這些通訊系統的那座橋。這也是你完全可以替自己另一隻代理動手實作的東西。

🧠 記憶的三種型態

記憶在前面好幾段都順帶提過，但它值得有自己的章節。Hermes 的記憶，其實以三種不同型態存在。

① Markdown 檔

就是前面提過的 soul.md、memory.md、user.md（後兩者放在記憶目錄裡）。它們會永遠被附在系統提示之後，接進上下文視窗。

② SQLite 全文逐字稿

Hermes 有能力喚回先前的對話，而這些對話當然不是以 Markdown 檔的形式存的。它另外有一個 SQLite 資料庫，把你和它每一次互動的每一則訊息都記下來。裡頭有很多張表、很多列、很多資料模型，但本質上都是同一件事的不同呈現方式——也就是所有工作階段的完整逐字稿。每段工作階段都有自己的 ID，閘道要接續對話時，也正是從這裡把對應的工作階段撈出來。

💡 還有一張「純文字表」

很有意思的是，它另外維護一張只放純文字的表，裡面只有所有對話的文字內容——目的是讓你能方便地對它做相似度搜尋。有了這層 SQLite 記憶，喚回往事就變得很實在。

③ 外部記憶（預設不開）

第三種是外部記憶，這一個預設不會幫你設定。它來自不同的外部供應商，而且為數不少：例如 Mem0、SuperMemory、Honcho 等。每一家運作方式都不同，但全都專精於「替代理提供記憶／智慧」這件事：

供應商大致做法
Mem0大致以相似度搜尋之類的方式運作。
Honcho採用另一套不同的機制。
SuperMemory每個回合後都要把整段對話歷史送過去，再由它用一個 LLM 從中萃取出該記住的記憶。

大多數人其實沒開外部記憶，但這裡仍建議你開——有不少支援的供應商是免費的，開了之後能大幅改善 Hermes 向你學習的能力。

🤓 一個實用小細節：它在「第一則之後」才查外部記憶

外部記憶設好後，它不是每則訊息都查，而是在第一則訊息之後才查。當你開啟一個新話題、代理回了你第一句、它已經大致知道你在講什麼，這時它才去查外部記憶，試著猜你下一個問題會是什麼——很像人被問問題時，先答出來，同時在腦中翻找過去相關的對話。所以實務上：如果第一次它沒想起某件事，你可以在第一則訊息先描述你想回憶的東西，再用一則追問，這時它就已經查過外部記憶了。

⏰ 排程工作（Cron Jobs）：每分鐘跳一次的 tick

如果你碰過伺服器，大概知道傳統的排程（cron）通常是一個每分鐘執行一次的系統程序，把當下那一分鐘該跑的任務撈出來執行。Hermes 大致也是這樣，但有個重點：Hermes 的 cron 並不綁在伺服器的 cron 程序上。它是自己的一個迴圈，每分鐘呼叫一個叫 tick 的函式——每分鐘跳一下，撈出你排的工作清單，當場執行。

排程工作，就是讓你能交代 Hermes 這類事情：「每天早上寄一封 AI 最新消息給我」「每天在 Slack 對社群發一則新進展」「每週五寄一封進度信給我主管」。它讓你能把任務自動化在某個明確時刻，並持續每天／每週／每月地跑下去。

⚠️ 文件說存在 SQLite，實際上是 JSON

這裡有個容易踩雷的落差：文件說排程工作存在 SQLite，但實際翻查 SQLite 資料庫並找不到，程式碼看起來也沒從 SQLite 拉。真正的情況是——排程工作以純 JSON 儲存，路徑在 ~/.hermes/cron/jobs.json。所有排程工作連同各自的提示、該做什麼，全列在這個檔裡；每次 tick，Hermes 就讀這個 JSON、判斷現在要不要跑某個工作。

在同一個 cron 目錄底下，還有一個 output 目錄，裡面是一堆以工作 ID 命名的子目錄；每個工作 ID 之下，存著該工作每一次執行的 Markdown 紀錄。整件事就這麼直白：一個每分鐘跑一次的迴圈，每分鐘檢查 jobs.json，有到點的就執行，再把結果寫進那次執行的 Markdown 檔。

🤓 小知識：cron 通知是送到你的「主要平台」

cron 不會自動在 Telegram 上幫你發訊息。它會把通知送到你的主要（home）通訊平台。設定閘道時（例如 Telegram、Discord、Slack），它會問你要不要把某個閘道、某個使用者 ID 設成這個閘道的「home」。之後每當排程工作執行，通知就會送到你設定的這個主要整合上。也就是說，cron 並不是叫代理去呼叫「送訊息」工具，而是由系統層級直接發一則通知給你。
