檔案系統技術百科：從 FAT32 到 ZFS 的演進之路

深入探討現代儲存格式的日誌功能、資料保護與效能表現，掌握不同作業系統的核心儲存技術

📂 檔案系統是什麼？

檔案系統就像是儲存裝置的「圖書管理員」。當你對硬碟 (HDD)、固態硬碟 (SSD) 或隨身碟進行格式化時，作業系統會將儲存空間劃分為無數個虛擬隔間，稱為「叢集 (Clusters)」，並建立一份詳細的索引表來追蹤檔案位置與可用空間。

不同的作業系統 (Windows, macOS, Linux) 與用途，催生了各種不同的檔案系統標準。選擇正確的格式，不僅影響跨平台相容性，更決定了單一檔案的大小上限與資料安全性。

NTFS

New Technology File System

📋 誕生： 1993 (Win NT 3.1)

💻 主場： Windows 系統碟

🚀 最大單檔： 16 TB（預設 4KB 叢集）～ PB 級（視版本與設定）

🔒 特色： 日誌式、權限管理

Windows 的絕對統治者

為了突破 FAT 的限制，微軟推出了 NTFS。現代 Windows 系統必須安裝在 NTFS 磁碟上。它支援日誌功能 (Journaling)，能在系統崩潰或斷電後快速恢復資料一致性，這點是 FAT32 所沒有的。

跨平台支援的痛點

雖然 NTFS 在 Windows 上強大且安全（支援檔案權限、加密），但在其他平台上卻有點「水土不服」。macOS 預設對 NTFS 只有「唯讀」權限，無法寫入資料；Linux 雖然支援，但有時需額外設定。這使得 NTFS 較不適合作為 Mac 與 PC 之間的傳輸媒介。

XFS

Extended File System (SGI)

📋 誕生： 1993 開發、1994 出貨 (IRIX 5.3)

🏢 主場： 企業級伺服器、大型儲存陣列

🚀 特色： 高效能大檔處理、極佳平行 I/O

吃粗飽的大檔高手

如果 ext4 比較像「萬用家用款」，XFS 就是「專門搬大型影片與資料庫」的重型機具：它是 64 位元日誌式檔案系統，對超大容量磁碟與單一超大檔案都有良好效能與延展性。在需要大量循序寫入或平行讀寫的情境（如影片剪輯伺服器、備份儲存庫），XFS 通常比 ext4 更能發揮硬體效能。

超大容量，但不擅長碎碎念

XFS 擅長的是「少量、超大檔」的工作型態，如果是儲存數以百萬計的小檔案，有時調校不佳會出現效能劣化。此外，XFS 的線上縮小檔案系統仍不支援，規劃容量時要有「只會越來越大」的心理準備，因此常搭配 LVM（Logical Volume Manager，邏輯磁碟區管理）或硬體 RAID 一起使用。

FAT32

File Allocation Table 32

📋 誕生： 1996 (Win 95 OSR2)

🤝 相容性： 極高 (Win/Mac/Linux)

🚫 最大單檔： 4 GiB − 1（約 4 GB）

💾 分割區上限： 常見 2 TB（512B 磁區），4K 磁區可更大

歷史悠久的通用標準

FAT 家族 (FAT12, FAT16, FAT32) 是最經典的檔案系統。FAT32 至今仍活躍於 USB 隨身碟與記憶卡，原因是它擁有幾乎完美的跨平台相容性 —— 從電腦、遊戲機到汽車音響都能讀取。

致命的 4 GB 限制

FAT32 最大的弱點是無法儲存超過 4GB 的單一檔案。如果你嘗試將一部高畫質電影或大型 ISO 映像檔複製到 FAT32 隨身碟，系統會直接拒絕。此外，Windows 的圖形化格式化工具長期只提供 FAT32 到 32GB，在較新的 Windows 11 測試版中，命令列格式化已可放寬到 2TB（仍視版本而定）。

HFS+

Hierarchical File System Plus

📋 誕生： 1998 (取代舊 HFS)

🍎 主場： 舊版 macOS、部分外接碟

🕰 角色： APFS 問世前的蘋果主力檔案系統

APFS 出現前的蘋果王者

在 2017 年 APFS 正式登場之前，所有 Mac 上的系統碟與多數外接硬碟幾乎清一色都是 HFS+ 格式（又稱 Mac OS Extended）。它支援長檔名、Unicode、日誌功能，也能處理大型磁碟與檔案，是當年從「經典 Mac OS」跨入 OS X 時代的重要基礎建設。

時代走到尾聲，仍在角落服役

雖然新版本 macOS 已全面改用 APFS，但你仍會在舊 Mac、舊 Time Machine 備份碟或與 Windows 共用的外接硬碟上看到 HFS+ 的身影（搭配第三方驅動）。相較於 APFS 的快照與原生加密，HFS+ 在現代功能上顯得老態，卻依舊扮演著「相容舊系統」的重要橋樑。

exFAT

Extended FAT

📋 誕生： 2006

⚡ 最佳化： 快閃記憶體 (Flash)

🔄 跨平台： Win/Mac 讀寫

📹 應用： 大容量記憶卡、隨身碟

為現代傳輸而生

exFAT 是 FAT32 的現代化繼任者，專為快閃記憶體最佳化。它採用 64 位元描述檔案大小，理論可達 EB 等級（實務受磁碟區與實作限制，常見整理為 PB 級），同時被 SD 協會選為 SDXC 記憶卡的標準格式。它是 4K 影片錄製和高容量隨身碟的最佳拍檔。

最佳的跨平台橋樑

exFAT 是目前Windows 與 macOS 之間共用外接硬碟的最佳選擇。雙方都能原生支援讀寫，無需安裝與安裝驅動程式。雖然它沒有 NTFS 的日誌保護功能（在現代 SSD 上，因控制器與磨耗平均機制成熟，日誌對壽命的影響通常不會成為主因），但在可攜式儲存裝置上，便利性勝過一切。

ZFS

Zettabyte File System

📋 誕生： 2006 (Sun Microsystems)

🖥 主場： NAS、伺服器、FreeBSD

🚀 最大單檔： 16 EB

🛡 特色： 資料驗證、RAID-Z、快照

檔案系統與磁碟管理的終極結合

ZFS (Zettabyte File System) 最初由 Sun Microsystems 開發，現由 OpenZFS 專案維護。它整合了檔案系統與磁碟區管理 (Volume Manager)，能直接控制硬體儲存陣列，透過 RAID-Z 提供資料冗餘保護。

位元衰減的剋星

ZFS 對每一個資料區塊都進行總和檢查碼 (Checksum) 驗證，能主動偵測「位元衰減 (Bit rot)」——這種儲存媒體上無聲無息的資料損壞。若 pool 設定了鏡像或 RAID-Z 等冗餘，ZFS 可自動以正確副本修復損壞資料。對於 NAS、備份伺服器與企業級儲存環境，ZFS 是資料完整性的最佳守護者。目前支援 Linux、FreeBSD 與 TrueNAS。

Btrfs

B-tree File System

📋 誕生： 2007 開發、2009 進入主線 (Linux 2.6.29)

🐧 主場： Linux 系統碟、伺服器

🧩 特色： 快照、壓縮、資料驗證與自動修復

Linux 的現代萬能工具箱

Btrfs 可以把它想像成「內建備份與版本控制」的檔案系統：每次建立快照 (Snapshot)，就像幫整個系統按下一次「時間點存檔」，崩潰時可以快速回到之前的狀態。它支援透明壓縮、子磁碟區 (Subvolume)、線上擴充與縮小容量，甚至可以在同一個檔案系統裡做 RAID，對家庭 NAS 與實驗環境非常友善。

資料完整性的守門員

與傳統 ext4 不同，Btrfs 對資料與中繼資料都計算總和檢查碼 (Checksum)，一旦偵測到「位元悄悄爛掉」的情況，若設定了 RAID1 等 replicated profile（有可用副本），可自動修復。這也代表它比較吃 RAM、調校沒做好的話效能可能略輸 ext4。另要注意，對某些 NOCOW 檔案可能隱含關閉資料驗證（NODATASUM），會影響資料完整性保護能力。

ext4

Fourth Extended File System

📋 誕生： 2008 (Linux Kernel 2.6.28)

🐧 主場： Linux 系統碟

🚀 最大單檔： 16 TB

🔧 特色： 日誌式、高穩定性

Linux 世界的萬用首選

Extended File System 家族歷經 ext2、ext3 到目前的 ext4，是絕大多數 Linux 發行版的預設系統碟格式。它繼承了 ext3 的日誌功能，同時大幅提升最大檔案與磁碟區容量限制，並改善了大型目錄的效能表現。

穩定為王，跨平台有限

ext4 以穩定、成熟著稱，是伺服器與桌面 Linux 的可靠選擇。然而，Windows 與 macOS 均無原生支援，若需在這些系統上讀取 ext4 磁碟，必須安裝第三方驅動程式（如 Ext2Fsd、Paragon ExtFS）。

F2FS

Flash-Friendly File System

📋 誕生： 2012 merge、2013 release (Linux 3.8)

📱 主場： Android、eMMC / UFS / SD 卡

⚡ 特色： 專為 Flash 設計，降低寫入放大、延長壽命

為快閃記憶體量身打造

傳統的 ext4、XFS 都是為硬碟時代設計，放到快閃記憶體上用得「還行但不優」，這就是 F2FS 出現的原因。它採用類似日誌結構檔案系統 (LFS, Log-Structured File System) 的設計，透過聰明的區塊配置方式減少隨機寫入與垃圾回收成本，讓 SSD、eMMC 在手機與嵌入式裝置上跑得更順也更耐用。

Android 與嵌入式的隱形主角

不少 Android 裝置與路由器、單板電腦的系統分割區，背後其實就是跑 F2FS，但一般使用者幾乎不會特別察覺。在桌面 Linux 上，你也可以把特定 SSD 分割區格式化成 F2FS，不過桌面發行版預設仍以 ext4 為主，F2FS 仍偏向「懂調校的人」會主動選用的選項。

ReFS

Resilient File System

📋 誕生： 2012 (Windows Server 2012)

🖥 主場： Windows Server、資料中心

🛡 特色： 自動錯誤偵測與修復、高可靠性

為資料中心而生的「韌性」檔案系統

顧名思義，ReFS 的重點不是華麗功能，而是「韌性」：它內建資料完整性串流 (Integrity Streams) 與背景檢查機制，能在讀取時發現靜默損壞，搭配 Storage Spaces 時，可以自動從鏡像或同位資料中修補。對動輒數十 TB、甚至 PB 級別的企業檔案伺服器與備份系統來說，這種「壞掉了自己修好」的能力比單純效能更關鍵。

桌面版本看得到，摸不太到

在某些 Windows 版本中，你可能在系統裡隱約看過 ReFS 的相關選項，但一般桌面使用者幾乎無緣直接把系統碟格式化成 ReFS。微軟近年來逐步把 ReFS 的建立權限收緊到特定版本與伺服器 SKU，讓它保持在「專業 IT 與資料中心」的定位，而不是拿來格式化家用筆電 C 槽。

APFS

Apple File System

📋 誕生： 2017 (macOS High Sierra)

🍎 主場： macOS / iOS / iPadOS 系統碟

🚀 最大單檔： 8 EB (理論值)

🔐 特色： 原生加密、快照、空間共享

為 SSD 與蘋果生態量身打造

Apple 於 2017 年推出 APFS 取代老舊的 HFS+，專為固態儲存裝置 (SSD) 最佳化。它支援原生全磁碟加密、快照 (Snapshot)、空間共享 (Space Sharing)，以及檔案與目錄的瞬間複製 (Clone)，讓 Time Machine 備份與系統更新更加高效。

封閉生態的代價

APFS 是蘋果生態的核心，但這也意味著Windows 完全無法原生讀取 APFS 磁碟。若需要跨平台共用外接硬碟，APFS 並非好選擇；但對於純 Mac / iPhone / iPad 使用者，它是目前最先進的系統碟格式。

💡 怎麼選？懶人包結論

  • ✔ 系統碟： 乖乖用 OS 預設 (Win 用 NTFS, Mac 用 APFS)。
  • ✔ 小隨身碟 (USB 2.0 / < 32GB)： 選 FAT32，相容性無敵。
  • ✔ 大隨身碟 / 記憶卡： 選 exFAT，避開 4GB 限制。
  • ✔ 外接硬碟 (Win/Mac 共用)： 優先推薦 exFAT（最通用的原生讀寫選擇）。

⚠️ 真實慘痛經驗：4 GB 陷阱

曾有一次，我為了單板電腦下載了一個「作業系統映像檔 (Image)」。我試著在隨身碟上用 WinZip 解壓縮，失敗！換 7-Zip，再失敗！換了一台電腦，還是失敗！

最後我才恍然大悟：「因為我的隨身碟是 FAT32 格式！」

雖然壓縮檔本身沒問題，但解壓縮後的檔案剛好超過 4GB。受限於 FAT32 的先天限制，檔案寫入到 4GB 時就會被迫中斷。這個教訓告訴我們：在高畫質影音與大檔時代，檢查檔案系統是基本功。
