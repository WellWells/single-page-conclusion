技術摘要

Meta 近期對 Windows 11 版本的 WhatsApp 進行了強制更新，終止了原本基於 WinUI (Windows UI Library) 的原生應用程式支援，轉而採用基於 WebView 的架構。這項改動導致記憶體佔用量暴增超過 10 倍。這不僅僅是一個單一軟體的更新失敗，更標誌著大型科技公司對桌面端體驗的放棄，以及軟體工程品質在「利潤至上」思維下的全面崩壞。

💾 記憶體怪獸的誕生

從輕量化原生應用到資源黑洞的演變

WhatsApp 的 Windows 版本經歷了一個諷刺的循環。最初它是基於 Electron (跨平台桌面應用框架) 構建，因資源消耗巨大而為人詬病。隨後，Meta 推出了基於原生 WinUI 的版本，大幅優化了效能與系統整合度。然而，現在他們又走回頭路，將其變回一個基於 Chromium 核心的網頁殼層（Shell）。

⚰️ 原生開發的黃昏

為何市值 1.5 兆美元的企業不願寫好軟體？

這是一個殘酷的商業現實：原生開發正在被系統性地拆解。對 Meta 這樣一家在海底鋪設光纖電纜、擁有無限資源的巨頭來說，維護一個 Windows 用戶端在技術上毫無難度。真正的問題在於「動機」。

「如果 WhatsApp 都可以這麼爛，那我們為什麼不行？」

Windows 桌面端無法帶來顯著的使用者增長或投資人興趣。當行業領頭羊開始為了節省成本而犧牲使用者體驗（UX, User Experience），這將成為業界的新常態。我們將看到更多公司裁撤原生開發團隊，轉而將資源投入到維護單一的 Web 程式碼庫，再透過充滿義大利麵式程式碼（Spaghetti Code）的 JavaScript 框架硬塞給所有平台。

📱 平台優先級的轉移

Desktop OS 已淪為瀏覽器的啟動器

我們必須承認一個趨勢：軟體開發現在是 Mobile First，Web Second，至於桌面作業系統（Desktop OS）充其量只是事後諸葛。現代使用者的使用習慣也支撐了這一點——打開筆電的第一件事，通常就是打開瀏覽器。

這直接衝擊了軟體工程師的就業市場。專精於特定平台（如 WPF, macOS Native）的職缺將減少，市場將極度傾斜向「全端通用型工程師」（Full-stack Generalists）。這驗證了著名的 Atwood 定律 (Atwood's Law)：

" 任何可以用 JavaScript 編寫的應用程式，最終都會用 JavaScript 編寫。 "

現在看來，只要你心臟夠大顆，確實「所有」應用程式都可以用 JavaScript 重寫——即使代價是巨大的效能損耗。

🏗️ 文明的技術債

Jonathan Blow 的預言與現代軟體的品質危機

資深開發者 Jonathan Blow 曾在其演講《Preventing the Collapse of Civilization》中提出警告：現代軟體品質低落、脆弱，且正在累積文明規模的技術債 (Technical Debt)。

  • ⚠️ 糟糕的抽象層：我們在搖搖欲墜的地基上堆疊框架。
  • ⚠️ 缺乏離線能力：斷網即癱瘓的軟體成為常態。
  • ⚠️ 硬體紅利被吃光：硬體變強不是為了讓軟體更快，而是為了讓開發者可以寫出更沒效率的程式碼。

過去開發者能在三週內寫出一個作業系統的核心；現在我們有敏捷開發（Agile）、Scrum、Jira、Confluence、Standups、Retrospectives... 擁有一切繁文縟節與管理流程，卻交付不出一個流暢的聊天視窗。

🔮 結論：工程師的未來

這個故事的結局雖然苦澀，但對開發者而言卻有一線生機。企業追求「快」與「便宜」的決策路徑已經固化，軟體品質不再是優先考量。然而，這種由糟糕程式碼與 AI 生成的垃圾內容（Slop）所堆砌出的軟體世界，將需要大量的維護與「保母」照看。

軟體工程師的工作在未來數十年內依然安全——雖然我們的工作可能不再是創造偉大的產品，而是永無止境地清理這些數位廢棄物。這不是世界上最棒的工作，但它是誠實的勞動。
